Transform Deep Dive

Every transform shorthand you've used is secretly matrix multiplication — and the order you write functions in changes the result.

1. Theory

rotate(), translate(), and scale() aren't independent mechanisms — each one compiles to a 4x4 transformation matrix, and transform: a() b() c() multiplies those matrices together. matrix3d() is that resulting matrix spelled out explicitly as 16 numbers, rather than a separate feature.

Matrix multiplication is not commutative, so the order functions are listed in changes the outcome: rotate(45deg) translateX(120px) first rotates the coordinate system, then translates along the now-rotated axis (the element moves diagonally); translateX(120px) rotate(45deg) translates first, then rotates around the already-shifted origin — a different final position entirely, from the exact same two function calls.

2. Browser internals

Regardless of whether a transform is written as a function list (rotate() translate()) or as an explicit matrix3d(), the browser resolves it down to the same matrix before handing it to the compositor — there's no performance difference between the two forms, only a readability one. All three of this chapter's demos stay entirely within transform, so — unlike Phase 5's SVG chapters, where several techniques genuinely forced paint or layout — everything here remains composite-only.

3. Step-by-step implementation

Step 1 (Demo 1) — the same two function calls, reversed, transitioned from rest on a button click:

.order-demo__box--rotate-first.is-transformed {
  transform: rotate(45deg) translateX(120px);
}
.order-demo__box--translate-first.is-transformed {
  transform: translateX(120px) rotate(45deg);
}

Step 2 (Demo 2) — a rotateZ + uniform scale + translateX composed by hand into the 16 numbers matrix3d() expects, column by column:

const cos = Math.cos(angle);
const sin = Math.sin(angle);
const values = [
  scale * cos, scale * sin, 0, 0,
  -scale * sin, scale * cos, 0, 0,
  0, 0, 1, 0,
  tx, 0, 0, 1,
];
box.style.transform = `matrix3d(${values.join(', ')})`;

Step 3 (Demo 3) — a parent with perspective gives its 3D children a shared vanishing point; transform-style: preserve-3d on the rotating group keeps the cards' individual translateZ positions intact as the whole group rotates:

.stack-demo__stage {
  perspective: 800px;
}
.stack-demo__group {
  transform-style: preserve-3d;
  animation: stack-rotate 6s linear infinite;
}
.stack-demo__card {
  transform: translateZ(var(--card-offset));
}

Switching that same group to transform-style: flat collapses each card's translateZ into the group's own 2D plane before the group's rotation is even applied — the cards visually flatten into a single overlapping stack instead of keeping their fanned 3D arrangement.

6. Common mistakes

  • Assuming transform function order doesn't matter — as Demo 1 shows directly, the exact same two function calls in reversed order produce two different final positions.
  • Applying transform-style: preserve-3d to the 3D-positioned children themselves instead of their shared parent — it needs to live on the element whose children should keep their individual 3D positions as a group, not on each child individually.
  • Forgetting perspective entirely (on the parent, or as a perspective() function) — 3D rotations still apply without it, but render as a flat, undistorted squash instead of true depth.

7. Performance considerations

Every demo in this chapter only ever touches transform — whether written as a function list or as an explicit matrix3d(), the browser resolves both to the same compositor-ready matrix, so this stays composite-only regardless of how complex the composition is. This is a deliberate contrast with Phase 5's SVG chapters, several of which genuinely forced paint or layout — nothing here does.

8. Accessibility considerations

Demo 1's trigger is a real <button>, fully keyboard-operable; the two boxes it animates are decorative with no unique information. Demo 2's sliders are native <input type="range"> elements, already keyboard-adjustable via arrow keys with no extra work needed — the same built-in accessibility this site's Transform Variants chapter (Phase 2) relies on for its own sliders. Demo 3's toggle is a native checkbox; the rotating card stack and the perspective-comparison boxes are both marked aria-hidden="true" since they're purely illustrative and convey nothing beyond what the surrounding prose already states.

9. Mobile considerations

All three demos work identically on touch: Demo 1's button taps like any button, Demo 2's range inputs are natively touch-draggable, and Demo 3's checkbox toggles via tap. Nothing in this chapter depends on hover or a mouse-only input.
