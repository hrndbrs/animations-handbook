Timing & Easing

Three different ways to shape motion over time — a physical simulation, an authored curve, and offset sequencing — and when to reach for each.

1. Theory

A spring is physically simulated: it has no fixed duration, it reacts continuously to wherever its target is, and it's naturally interruptible — drag it away mid-motion and it just starts springing from the new position, no restart logic needed. A cubic-bezier curve is the opposite: an authored, fixed-duration shape you draw once and reuse everywhere, the same curve producing identical motion every time it plays. Staggering is a different axis entirely — it doesn't shape any one element's motion, it shapes the *offsets between* multiple elements' identical motions.

Springs suit physically-motivated, interruptible interactions (drag-and-release, this chapter's Demo 1). Cubic-bezier suits fixed-duration authored transitions where you want a specific, repeatable feel (Demo 2). Stagger suits sequencing many elements without hand-writing a delay for each one (Demo 3).

2. Browser internals

A spring has to be simulated step by step in JS — there's no CSS primitive for "settle toward a target with this stiffness and damping," so Demo 1 integrates the spring's acceleration into velocity into position every frame, the same requestAnimationFrame-loop pattern this handbook has used since Phase 1's Canvas chapter. A cubic-bezier curve, by contrast, is natively understood by CSS's transition-timing-function — Demo 2 plays the identical curve two ways at once (a native CSS transition, and a JS loop evaluating the same curve via utils/easing.js's cubicBezier) so you can confirm both produce the same motion from the same four numbers.

3. Step-by-step implementation

Step 1 (Demo 1) — one spring-integration step, called once per frame with the previous state:

import { stepSpring } from '../../utils/spring.js';

function animateSpring(timestamp) {
  const dt = Math.min((timestamp - lastFrameTime) / 1000, 0.05);
  const next = stepSpring({ position, velocity, target: 0, stiffness, damping, dt });
  position = next.position;
  velocity = next.velocity;
  box.style.transform = `translate(${position.x}px, ${position.y}px)`;
  if (!settled) requestAnimationFrame(animateSpring);
}

Step 2 (Demo 2) — the exact same four control-point numbers drive a native CSS transition and a JS loop side by side:

.bezier-demo__box--css.is-playing {
  transform: translateX(200px);
  /* transition-timing-function set inline from the same p1/p2 as the JS below */
}
import { cubicBezier } from '../../utils/easing.js';

const ease = cubicBezier(p1.x, p1.y, p2.x, p2.y);
function step(timestamp) {
  const progress = Math.min((timestamp - startTime) / 1000, 1);
  jsBox.style.transform = `translateX(${ease(progress) * 200}px)`;
  if (progress < 1) requestAnimationFrame(step);
}

Step 3 (Demo 3) — each item's transition-delay is computed from its index, either linearly or shaped by smoothstep:

import { smoothstep } from '../../utils/math.js';

function getLinearDelay(index) {
  return (index / (count - 1)) * totalMs;
}
function getEasedDelay(index) {
  return smoothstep(0, 1, index / (count - 1)) * totalMs;
}

5. Exercises

  1. Set Demo 1's damping slider to its minimum and observe the box oscillate for much longer before settling — an underdamped spring.
  2. Drag Demo 2's second handle so its y-value goes above 1.0 or below 0.0, and watch both boxes overshoot past their target before settling — a real, valid cubic-bezier technique, not a bug.
  3. Change Demo 3's getEasedDelay to use 1 - smoothstep(0, 1, 1 - t) (an eased curve shaped the other way) and compare how the item spacing changes.

6. Common mistakes

  • Using a spring with damping at or near zero, expecting it to settle — with too little damping relative to stiffness, it oscillates for a very long time (or effectively forever) instead of coming to rest.
  • Assuming cubic-bezier control points must stay within the 0-1 range — they don't; y-values outside that range are exactly what produce overshoot/bounce easing, as Demo 2's Exercise 2 shows directly.
  • Giving every staggered item the same transition-delay by mistake (e.g. computing it outside the per-item loop) — nothing appears staggered at all, since every item then starts at the same time.

7. Performance considerations

All three demos only ever animate transform or opacity, so all stay composite-only regardless of which timing mechanism drives them. The real cost difference between the three techniques is JS computation per frame, not render cost: a spring integration step and a Newton-Raphson bezier solve are both cheap for a handful of elements, but a page driving hundreds of simultaneous spring-animated elements would want to batch that math rather than run 300+ independent requestAnimationFrame loops.

8. Accessibility considerations

Demo 1's drag-and-release has no keyboard equivalent — dragging is inherently a pointer gesture — but the circle is purely decorative with no functional information gated behind it, the same reasoning this handbook's Phase 4 pointer-driven chapters already established. Demo 2's draggable handles are similarly pointer-only, but the "Play both" button is a real, keyboard-operable trigger that lets a keyboard-only user see the currently-set curve play even without ever dragging a handle. Demo 3's reveal is a real button; the staggered items themselves are decorative.

9. Mobile considerations

Demo 1 and Demo 2 both use Pointer Events (pointerdown/pointermove/pointerup), which unify mouse, touch, and pen input — both demos' dragging works identically via touch, and touch-action: none on the draggable elements prevents the browser's default scroll/zoom gestures from fighting the drag. Demo 3's button works via tap exactly as it does via click.
