Canvas Particle System

The imperative engine: you compute and draw every frame yourself.

1. Theory

Everything up to this chapter has been declarative — you set a CSS end state and the browser interpolated the frames. Canvas is the opposite model: <canvas> is an immediate-mode drawing surface with no retained scene graph. Nothing you draw "exists" as an object the browser tracks — each frame you clear the canvas and redraw everything from scratch based on your own in-memory state.

This exists for cases CSS/DOM animation can't reach: particle counts in the hundreds or thousands, per-pixel effects, physics simulations — anything where creating one DOM element per visual unit would be far too much overhead for the browser to lay out, style, and composite individually.

2. Browser internals

A <canvas> element is a single compositor layer. No matter how many shapes you draw inside it, the browser only ever has to composite one bitmap for the whole canvas — the cost lives entirely in your own per-frame JavaScript and the 2D context's rasterization work, not in browser layout or per-element paint.

requestAnimationFrame (rAF) schedules your callback to run right before the browser's next repaint, synced to the display's refresh rate (typically 60Hz, sometimes 120Hz+). This is different from a setInterval loop, which runs on a fixed timer with no relationship to the browser's actual paint cycle — rAF callbacks that miss a frame budget just push to the next repaint instead of stacking up queued work.

3. Step-by-step implementation

Step 1 — the canvas element and its 2D context:

<canvas id="particle-canvas" width="600" height="360"></canvas>
const canvas = document.getElementById('particle-canvas');
const ctx = canvas.getContext('2d');

width/height attributes set the canvas's drawing-buffer resolution (its coordinate space) — this is distinct from its CSS display size, which is handled separately in particles.css.

Step 2 — a Particle class holding each particle's home position and current position, using lerp/clamp from the shared math utilities to smooth its motion:

import { lerp, clamp } from '../../utils/math.js';

class Particle {
  constructor() {
    this.reset();
  }
  reset() {
    this.baseX = randomBetween(0, canvas.width);
    this.baseY = randomBetween(0, canvas.height);
    this.x = this.baseX;
    this.y = this.baseY;
    this.radius = randomBetween(1, 3);
  }
  update(pointer) {
    const dx = pointer.x - this.x;
    const dy = pointer.y - this.y;
    const distance = Math.hypot(dx, dy);
    const influence = clamp(1 - distance / 160, 0, 1);
    const pushX = influence * -dx * 0.05;
    const pushY = influence * -dy * 0.05;
    this.x = lerp(this.x, this.baseX + pushX, 0.08);
    this.y = lerp(this.y, this.baseY + pushY, 0.08);
  }
  draw() {
    ctx.beginPath();
    ctx.arc(this.x, this.y, this.radius, 0, Math.PI * 2);
    ctx.fillStyle = 'rgba(79, 70, 229, 0.85)';
    ctx.fill();
  }
}

Step 3 — track the pointer position with a pointermove listener, registered as passive since it never calls preventDefault():

canvas.addEventListener('pointermove', (event) => {
  const rect = canvas.getBoundingClientRect();
  pointer.x = ((event.clientX - rect.left) / rect.width) * canvas.width;
  pointer.y = ((event.clientY - rect.top) / rect.height) * canvas.height;
}, { passive: true });

Step 4 — the render loop: clear, update every particle, draw every particle, schedule the next frame:

function animate() {
  ctx.clearRect(0, 0, canvas.width, canvas.height);
  particles.forEach((particle) => {
    particle.update(pointer);
    particle.draw();
  });
  requestAnimationFrame(animate);
}

requestAnimationFrame(animate);

Note the loop calls requestAnimationFrame(animate) again from inside itself — this self-scheduling pattern is what keeps it running at the display's refresh rate indefinitely, instead of a fixed-interval timer.

5. Exercises

  1. Change PARTICLE_COUNT and observe the frame rate impact in DevTools' Performance panel at very high counts (try 2000+).
  2. Change EASE_TOWARD_TARGET (the third argument to lerp in update()) and observe how it changes the "springiness" of the drift.
  3. Add a color gradient based on each particle's distance from the pointer, using map() from utils/math.js to convert distance into a hue value.

6. Common mistakes

  • Forgetting ctx.clearRect() at the top of the loop — previous frames never get erased, leaving streaks.
  • Using setInterval instead of requestAnimationFrame — runs out of sync with the display's actual repaint, wasting work or looking stuttery.
  • Reading canvas.width/canvas.height as CSS pixels — they're the drawing-buffer resolution; mixing them up with the CSS display size (especially after scaling in `particles.css`) misplaces the pointer coordinates, which is why this demo explicitly rescales `clientX`/`clientY` through `getBoundingClientRect()`.
  • Allocating new objects (e.g. `{ x, y }`) inside the per-frame loop — creates GC pressure at 60fps; this demo reuses the same `Particle` instances and the same `pointer` object every frame instead.

7. Performance considerations

The loop's cost is O(particle count) per frame, entirely inside your own JS + the 2D context's fill/stroke calls — there's no browser layout or per-shape paint step to account for, since nothing here is a DOM element. The Particle instances and the pointer object are created once and mutated in place every frame, rather than reallocated, to avoid unnecessary garbage collection pauses during the animation.

8. Accessibility considerations

The canvas has an aria-label describing what it shows, since a canvas's pixel content is otherwise invisible to assistive technology. Because CSS's prefers-reduced-motion media query has no effect on a JS requestAnimationFrame loop, this demo checks matchMedia('(prefers-reduced-motion: reduce)') directly in JavaScript and renders a single static frame instead of starting the loop when the user has that preference set. 

const prefersReducedMotion = matchMedia('(prefers-reduced-motion: reduce)').matches;

if (prefersReducedMotion) {
  drawFrame();
} else {
  requestAnimationFrame(animate);
}

9. Mobile considerations

pointermove covers mouse, pen, and touch input uniformly (unlike separate mousemove/touchmove listeners). On a touchscreen there's no persistent "hover" position the way there is with a mouse, so particles simply drift back toward their baseX/baseY home positions when no active touch is present. Lower-powered mobile GPUs handle a canvas this size and particle count without difficulty, but PARTICLE_COUNT is a reasonable first thing to reduce if targeting older mobile hardware.
