Scroll Reveal with IntersectionObserver

Detecting viewport entry without a single scroll event listener.

1. Theory

The old way to reveal content on scroll was a scroll event listener that, on every fired event, called getBoundingClientRect() on every candidate element to check whether it had entered the viewport. That runs dozens of times per second during a scroll gesture, and each getBoundingClientRect() call can force a synchronous layout if styles are "dirty" (a forced reflow).

IntersectionObserver exists specifically to replace that pattern. You register a callback and a target element; the browser — not your code — tracks intersection with the viewport (or another ancestor) asynchronously, off the main thread's critical path, and only invokes your callback when the intersection state actually crosses a threshold you specify. No polling, no manual geometry math, no forced layout.

2. Browser internals

Because IntersectionObserver computations happen asynchronously and are batched by the browser, they don't compete with the main thread the way a naive scroll handler does. The animation this chapter triggers is still a plain CSS transition on opacity/transform — so once the observer flips the is-visible class, the actual animation work is the same composite-only path described in the CSS Transitions chapter.

3. Step-by-step implementation

Step 1 — the markup, a plain list:

<ul class="reveal-list__items">
  <li class="reveal-item">Item one</li>
  <li class="reveal-item">Item two</li>
  <li class="reveal-item">Item three</li>
</ul>

Step 2 — the resting and hidden states, both defined purely in CSS:

.reveal-item {
  opacity: 0;
  transform: translateY(24px);
  transition: opacity 0.6s ease-out, transform 0.6s ease-out;
}

.reveal-item.is-visible {
  opacity: 1;
  transform: translateY(0);
}

Step 3 — stagger each item's reveal by giving it an index-based transition-delay:

items.forEach((item, index) => {
  item.style.transitionDelay = `${index * 80}ms`;
});

Step 4 — create the observer and watch every item; once an item is 20% visible, mark it revealed and stop watching it:

const observer = new IntersectionObserver(
  (entries) => {
    entries.forEach((entry) => {
      if (entry.isIntersecting) {
        entry.target.classList.add('is-visible');
        observer.unobserve(entry.target);
      }
    });
  },
  { threshold: 0.2 }
);

items.forEach((item) => observer.observe(item));

threshold: 0.2 means the callback fires once 20% of the item's area is inside the viewport — tune this per design (a small threshold reveals items sooner, a large one waits until more of the element is visible).

5. Exercises

  1. Change threshold to 1 and observe how much longer items take to reveal.
  2. Add a rootMargin option to the observer so items reveal slightly before they enter the viewport.
  3. Remove the observer.unobserve(entry.target) call and also toggle is-visible off when entry.isIntersecting is false, to make the reveal repeat every time an item scrolls in and out.

6. Common mistakes

  • Using a scroll event listener with getBoundingClientRect() instead of IntersectionObserver — causes forced reflows and unnecessary main-thread work.
  • Forgetting to unobserve() a one-shot reveal — the observer keeps tracking elements that will never need to fire again.
  • Animating height or margin-top instead of transform: translateY() for the slide-in — forces layout on every frame of the transition.
  • Setting the initial opacity: 0 state only via a class that JS adds after load — causes a flash of fully-visible content before JS runs. Here it's set directly in the base CSS rule to avoid that.

7. Performance considerations

Intersection computation is handled by the browser off the critical rendering path, and the reveal animation itself only touches opacity/transform — composite-only, no layout thrash, regardless of how many items are in the list.

8. Accessibility considerations

Content is present in the DOM and readable by screen readers immediately — only its visual opacity/position animates, nothing is display: none or removed before reveal. The prefers-reduced-motion rule below collapses the transition to near-instant. 

@media (prefers-reduced-motion: reduce) {
  .reveal-item {
    transition-duration: 0.01ms;
  }
}

9. Mobile considerations

IntersectionObserver works identically for touch-driven scrolling; there's no :hover dependency. On low-powered devices, keeping the per-item transition to opacity/transform only (rather than adding blur filters or shadows to the transition) keeps the reveal smooth.
