CSS Transitions & Transform Basics

The declarative animation engine: change a value, let CSS animate the change.

1. Theory

A CSS transition animates a property from its old computed value to its new one whenever that value changes — on :hover, on a class toggle, on any style recalculation. You never write the in-between frames; the browser interpolates them. This is fundamentally different from JavaScript animation, where you compute every frame yourself.

transform is the property most worth animating this way. Unlike top/left/width/height, a transform change does not affect document layout — the element still occupies its original box, just painted somewhere else or at a different scale/rotation. That means animating transform can skip layout recalculation entirely.

Why this matters: it exists because early web animation used left/top, which forces the browser to recompute layout for every frame — expensive, and the reason old-web animations often looked janky. transition + transform was the platform's answer: a declarative, layout-free way to animate.

2. Browser internals

A browser's rendering pipeline has (roughly) three stages relevant here: Layout (compute every element's geometry), Paint (rasterize pixels), and Composite (assemble painted layers on the GPU).

Changing transform or opacity can skip Layout and Paint entirely and go straight to Composite, if the element already has its own compositor layer. Browsers promote an element to its own layer under several conditions — a running transform/opacity transition is one of them, since the browser knows in advance that only compositing will be needed. This is why transform/opacity are called the "cheap" properties to animate: they're the only two that can be composite-only.

Animating width, margin, top, or similar box-model properties forces Layout on every frame (a "reflow"), which cascades into Paint and Composite too — three stages of work per frame instead of one.

3. Step-by-step implementation

Step 1 — the markup. A <button> (keyboard-accessible by default) wraps two stacked faces:

<button type="button" class="flip-card" id="flip-card-trigger" aria-pressed="false">
  <span class="flip-card__inner">
    <span class="flip-card__face flip-card__face--front">Hover or click me</span>
    <span class="flip-card__face flip-card__face--back">Flipped!</span>
  </span>
</button>

aria-pressed="false" is the toggle's source of truth — both the visual flip and assistive-technology state read from it.

Step 2 — give the parent perspective, so the 3D rotation reads as a flip rather than a flat squash:

.flip-card-demo {
  perspective: 800px;
}

Step 3 — the two faces are absolutely positioned on top of each other, and the back face is pre-rotated 180deg so it lands right-side-up once the parent flips:

.flip-card__inner {
  transform-style: preserve-3d;
  transition: transform 0.6s cubic-bezier(0.65, 0, 0.35, 1);
}

.flip-card[aria-pressed="true"] .flip-card__inner {
  transform: rotateY(180deg);
}

.flip-card__face {
  position: absolute;
  inset: 0;
  backface-visibility: hidden;
}

.flip-card__face--back {
  transform: rotateY(180deg);
}

backface-visibility: hidden hides each face when it's turned away from the viewer, so you never see the back face's mirror image bleeding through the front face mid-flip.

Step 4 — the only JavaScript: toggle the source-of-truth attribute on click. CSS does the rest.

const trigger = document.getElementById('flip-card-trigger');

trigger.addEventListener('click', () => {
  const isPressed = trigger.getAttribute('aria-pressed') === 'true';
  trigger.setAttribute('aria-pressed', String(!isPressed));
});

Step 5 — as a concrete check that the CSS curve and a JS-side evaluator agree, this chapter also imports the cubicBezier evaluator from utils/easing.js and samples it at a few points using the same four control-point values as the CSS transition above:

import { cubicBezier } from '../../utils/easing.js';

const flipEasing = cubicBezier(0.65, 0, 0.35, 1);
const samples = [0, 0.25, 0.5, 0.75, 1]
  .map((t) => `t=${t} → ${flipEasing(t).toFixed(3)}`)
  .join('   ');
document.getElementById('bezier-sample-output').textContent = samples;

5. Exercises

  1. Change the transition's cubic-bezier() values and observe how the flip's acceleration curve changes.
  2. Add a second transition on box-shadow that grows the card's shadow while it's flipped, to practice animating more than one property at once.
  3. Make the flip trigger on :hover in addition to click, and consider why click-only might be the better choice for touch devices.

6. Common mistakes

  • Animating top/left instead of transform: translate() — forces layout every frame.
  • Forgetting backface-visibility: hidden — the back face shows through mirrored during the flip.
  • Putting perspective on the flipping element itself instead of its parent — the 3D effect won't read correctly.
  • Using a <div> with a click handler instead of a <button> — loses free keyboard accessibility and focus styling.

7. Performance considerations

This demo only ever animates transform, so it stays composite-only for its entire duration — no forced layout, no repaint of surrounding content. For a card this size that's not critical, but the same principle matters enormously once you're animating dozens of cards or a whole page section.

8. Accessibility considerations

The trigger is a real <button>, reachable and activatable by keyboard with no extra code. Its aria-pressed state is announced by screen readers as a toggle. The prefers-reduced-motion media query below shrinks the transition to effectively instant for users who've asked for reduced motion, while still changing state correctly. 

@media (prefers-reduced-motion: reduce) {
  .flip-card__inner {
    transition-duration: 0.01ms;
  }
}

9. Mobile considerations

There's no :hover dependency here — the demo is click/tap-driven, which works identically with mouse, touch, and keyboard (Enter/Space on the focused button). 3D transforms and backface-visibility are well supported on mobile WebKit/Blink, so no fallback is needed.
