Animated Icons

Three small, practical icon animations — a checkbox, a like button, a loading spinner — and the first appearance of SMIL, SVG's own built-in animation system.

1. Theory

SMIL (<animate>, <animateTransform>) is SVG's native, markup-only animation system — it predates both CSS animations and the technique of driving attributes from JS, and still ships in every evergreen browser. It can run entirely declaratively (no JS at all, as in Demo 3's SMIL spinner) or be triggered from JS via .beginElement() on an element with begin="indefinite" (Demo 2's SMIL like button). This chapter puts SMIL and CSS side by side on the exact same effect twice, so the tradeoffs are visible rather than abstract.

Demo 1 combines both of Phase 5a's core techniques into one real UI pattern: the checkbox's outer box "pops" via point interpolation (utils/svg-morph.js's interpolatePath, morphing between an inset and a full-size square), while its checkmark draws in via stroke-dashoffset (Path Drawing's technique) — driven by nothing more than the button's own aria-checked attribute in CSS.

2. Browser internals

SMIL's animateTransform type="scale" scales from the SVG coordinate system's origin (0, 0), not from an element's own visual center — there is no SMIL equivalent of transform-origin. Reaching a shape's own center requires wrapping it in a <g transform="translate(cx, cy)"> first, so the shape's local origin lands on its visual center before the scale is applied. type="rotate" sidesteps this by accepting an explicit center directly in its value ("angle cx cy"), which is why Demo 3's SMIL spinner needs no wrapper while Demo 2's SMIL like button does. CSS's transform-box: fill-box; transform-origin: center; (used by both CSS demos here) is the more convenient modern equivalent — one declaration instead of a markup restructure.

Critically, the sitewide prefers-reduced-motion CSS rule in shared/doc-page.css has no effect on SMIL animations at all, since SMIL isn't CSS and a CSS media query can't reach into it. Demo 3's SMIL spinner is paused explicitly via svgRoot.pauseAnimations() when reduced motion is preferred; its CSS twin is already covered by the sitewide rule.

3. Step-by-step implementation

Step 1 (Demo 1) — the checkmark draws in purely via CSS, reacting to the checkbox's own aria-checked attribute, no JS involved for this half:

.icon-demo__checkbox-check {
  stroke-dasharray: 100;
  stroke-dashoffset: 100;
  transition: stroke-dashoffset 0.3s ease;
}
.icon-demo__checkbox[aria-checked="true"] .icon-demo__checkbox-check {
  stroke-dashoffset: 0;
}

Step 2 (Demo 1) — the box "pops" by interpolating between two same-count point arrays, exactly like Phase 5a's Shape Morphing click demo:

const BOX_REST_POINTS = [[6, 6], [18, 6], [18, 18], [6, 18]];
const BOX_CHECKED_POINTS = [[2, 2], [22, 2], [22, 22], [2, 22]];

checkbox.addEventListener('click', () => {
  const isChecked = checkbox.getAttribute('aria-checked') === 'true';
  const from = isChecked ? BOX_CHECKED_POINTS : BOX_REST_POINTS;
  const to = isChecked ? BOX_REST_POINTS : BOX_CHECKED_POINTS;
  checkbox.setAttribute('aria-checked', String(!isChecked));
  // ...eased requestAnimationFrame loop calling
  // boxPath.setAttribute('d', interpolatePath(from, to, eased)) each frame
});

Step 3 (Demo 2) — the SMIL version needs a translate wrapper to scale from its visual center, and a JS-callable trigger:

<g transform="translate(12,12)">
  <circle cx="0" cy="0" r="8" fill="var(--color-accent)">
    <animateTransform id="like-smil-anim" attributeName="transform"
      type="scale" begin="indefinite" dur="0.4s"
      values="1;1.4;1" keyTimes="0;0.5;1" fill="remove" />
  </circle>
</g>
likeBtnSmil.addEventListener('click', () => {
  likeSmilAnim.beginElement();
});

Step 4 (Demo 2) — the CSS version needs no wrapper, only transform-box, and restarts itself via the standard remove/reflow/re-add pattern:

.icon-demo__like-dot--css {
  transform-box: fill-box;
  transform-origin: center;
}
.icon-demo__like-dot--css.is-pulsing {
  animation: like-pulse 0.4s ease;
}
@keyframes like-pulse {
  50% { transform: scale(1.4); }
}

Step 5 (Demo 3) — the SMIL spinner rotates continuously with no JS at all, since type="rotate" can specify its own center directly:

<animateTransform attributeName="transform" type="rotate"
  from="0 12 12" to="360 12 12" dur="1s" repeatCount="indefinite" />

5. Exercises

  1. Add a third checkbox state (indeterminate) with its own box-morph point array, sized between the rest and checked states.
  2. Change Demo 2's SMIL values from "1;1.4;1" to a 4-keyframe bounce and compare how much more verbose the SMIL markup becomes versus adding a CSS keyframe.

6. Common mistakes

  • Assuming the sitewide CSS prefers-reduced-motion rule silences SMIL animations too — it doesn't, since SMIL isn't CSS. Any SMIL animation on a real site needs its own explicit check, as Demo 3's spinner shows.
  • Applying animateTransform type="scale" directly to a circle without a translate wrapper — the circle visibly drifts toward the coordinate origin as it scales, instead of growing/shrinking in place.
  • Forgetting the reflow-forcing step before re-adding a CSS animation class on a rapid second click — without it, the browser may batch the class removal and re-addition into one recalculation and the animation silently never restarts.

7. Performance considerations

Demo 1's box-morph is a geometry change (like all of Phase 5a's shape morphing, not composite-only). Demos 2 and 3's pulse/spin are ordinary transform changes underneath — whether driven by SMIL or CSS, both stay composite-only, since neither touches layout or paint. The animation *system* (SMIL vs CSS) doesn't change this; only the *property* being animated does.

8. Accessibility considerations

Demo 1 is a real toggle with role="checkbox" and aria-checked, fully keyboard-operable and correctly announced by assistive tech regardless of whether its animation runs. Demos 2 and 3 are decorative reinforcement of state changes that already have other indicators (a like count would change elsewhere in a real UI; a spinner implies a loading state communicated some other way, e.g. aria-busy) — this chapter's simplified stand-ins don't wire that up since it's outside this chapter's scope, but a real implementation must. Both SMIL and CSS motion in this chapter respect reduced-motion preferences, per the Browser Internals section's explicit SMIL-specific handling.

9. Mobile considerations

All three demos are click/tap-driven or fully automatic — none depend on hover or a mouse-only input, so they behave identically on touch devices.
