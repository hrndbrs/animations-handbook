SMIL vs CSS vs JS Animation

The same effect — a stroke draws in, then the shape spins forever — built three ways, so the tradeoffs are visible instead of abstract.

1. Theory

This handbook has now used all three ways to animate SVG: SMIL (Animated Icons), CSS (everywhere), and JS (requestAnimationFrame, Path Drawing/Shape Morphing). None of them is strictly "best" — each wins on a different axis. This chapter builds the identical two-phase effect three times to make that concrete rather than asserted.

The clearest single difference is sequencing. SMIL can start one animation exactly when another finishes, by referencing its id: begin="draw.end". CSS has no equivalent — a second @keyframes animation meant to start after the first must use a hardcoded animation-delay matching the first animation's known duration, a coupling that silently drifts out of sync the moment either duration changes without the other being updated to match. JS has the opposite trade: it requires the most code (a manual timeline), but in exchange offers the most control — this chapter's JS panel is the only one with working Pause/Resume buttons, because pausing/resuming an arbitrary point in an animation is what a manual timeline gives you for free and SMIL/CSS do not, without extra plumbing this chapter doesn't build (SMIL does have pauseAnimations()/unpauseAnimations() on the SVG root, which pause every SMIL animation in that document at once — coarser than the JS panel's per-timeline control).

One more real gotcha, found while building this exact demo: begin="id.end" is standard, valid SMIL syntax (a "syncbase" reference), but this project found that current Chromium doesn't always reliably resolve it when it's the page's only such chained animation — the first animation's endEvent fires on schedule, but the dependent element's begin never resolves and it never starts. The lesson isn't "don't use syncbase timing"; it's that declarative sequencing still benefits from a JS fallback for reliability across engines. That's why the shipped SMIL panel pairs the declarative reference with an explicit listener: the rotation element is declared begin="indefinite", and a smil-draw endEvent listener calls .beginElement() on it in JS — belt and suspenders, not a workaround for broken syntax.

One persistent myth is worth correcting directly: SMIL is not deprecated. Chrome announced an intent to deprecate it in 2015, then reversed that decision in 2016 after developer pushback and the recognition that no adequate replacement existed for several of its capabilities. It remains implemented in every evergreen browser today.

2. Browser internals

All three panels animate the same two properties in the same order: stroke-dashoffset during the draw-in (a paint operation, per the Path Drawing chapter), then transform: rotate() during the spin (composite-only). The animation *system* driving those properties — SMIL, CSS, or a JS requestAnimationFrame loop — does not change what the browser has to do to render each property change; it only changes who is responsible for computing the next value and when.

3. Step-by-step implementation

Step 1 (SMIL panel) — the draw-in and the rotation are two separate <animate>/<animateTransform> elements. The rotation is declared begin="indefinite" rather than a smil-draw.end syncbase reference, because current Chromium doesn't always reliably resolve that reference when it's the page's only chained SMIL animation — so a JS listener on smil-draw's endEvent starts it explicitly instead:

<path id="smil-shape-path" class="comparison-demo__shape">
  <animate id="smil-draw" attributeName="stroke-dashoffset"
    from="100" to="0" dur="1s" fill="freeze" />
  <animateTransform id="smil-rotate" attributeName="transform" type="rotate"
    from="0" to="360" dur="1.5s" begin="indefinite" repeatCount="indefinite" />
</path>
document.getElementById('smil-draw').addEventListener('endEvent', () => {
  document.getElementById('smil-rotate').beginElement();
});

Step 2 (CSS panel) — the same two phases, but the rotation's start must be a hardcoded delay matching the draw's known duration:

.comparison-demo__shape--css {
  animation:
    comparison-draw 1s ease forwards,
    comparison-spin 1.5s linear infinite 1s; /* the "1s" delay must match the draw's own duration by hand */
}

Step 3 (JS panel) — a manual timeline tracks elapsed time across frames, computing the correct phase and value directly, which is also what makes pausing trivial (just stop accumulating elapsed time):

function tick(timestamp) {
  if (!isPaused) {
    elapsed += timestamp - lastFrameTime;
  }
  lastFrameTime = timestamp;
  render(); // sets stroke-dashoffset during the draw phase, then transform: rotate() after
  requestAnimationFrame(tick);
}

5. Exercises

  1. Change the CSS panel's draw duration from 1s to 2s without updating its rotation's hardcoded 1s delay, and observe the rotation starting visibly before the draw finishes — the exact drift this chapter's Theory section describes.
  2. Add a Pause/Resume pair for the SMIL panel using pauseAnimations()/unpauseAnimations() on its root <svg>, and compare its coarseness (it pauses every SMIL animation in that SVG at once) against the JS panel's single-timeline control.
  3. Add a fourth panel using the Web Animations API (element.animate(...)) — a fourth real option this chapter doesn't cover, sitting between CSS's declarative simplicity and JS's full manual control.

6. Common mistakes

  • Repeating the old "SMIL is deprecated" claim — it isn't; Chrome's 2015 deprecation intent was reversed in 2016, and it remains supported in every evergreen browser.
  • Assuming the sitewide CSS prefers-reduced-motion rule covers the SMIL panel — it doesn't (see Phase 5b's Animated Icons chapter); this chapter's SMIL panel is paused explicitly in JS instead.
  • Hardcoding a CSS animation-delay to sequence two animations, then changing one animation's duration later without updating the other's delay to match — the classic drift this chapter's first Exercise reproduces deliberately.
  • Relying solely on begin="id.end" for chained SMIL timing with no JS fallback — it's valid, standard syntax, but some engines (Chromium, as of this writing) don't reliably resolve syncbase-dependent timing when it's the only such chained animation active on a page; this chapter's SMIL panel pairs the declarative reference with a .beginElement() call in JS for that reason.

7. Performance considerations

All three panels animate the same properties (a paint-only draw phase, then a composite-only rotation) regardless of which system drives them — SMIL, CSS, and JS all reach the same rendering cost for identical property changes. The real cost difference between the three systems here is developer/maintenance cost (sequencing fragility, code volume), not render performance.

8. Accessibility considerations

All three shapes are purely decorative (aria-hidden="true") demonstrations of the technique, not functional content — no keyboard equivalent is needed for any of them. The JS panel's Pause/Resume buttons are real, keyboard-operable <button> elements; their disabled state (only one of the pair is ever actionable at a time) is reflected both visually and to assistive tech via the native disabled attribute, not a custom ARIA state.

9. Mobile considerations

All three panels run identically regardless of input device — none depend on hover or pointer position. The JS panel's Pause/Resume buttons work via tap exactly as they do via click.
