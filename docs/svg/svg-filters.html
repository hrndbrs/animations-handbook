SVG Filters & Effects

Blur, noise, and displacement — and why none of it can be driven by a CSS custom property.

1. Theory

An SVG <filter> is a small pipeline of primitive operations (feGaussianBlur, feTurbulence, feDisplacementMap, and others) applied to an element referenced via filter: url(#id). Each primitive has its own attributes — stdDeviation, baseFrequency, scale — but these live inside the filter's own internal graph, not on the styled element itself. CSS custom properties cannot reach into a referenced filter's primitive attributes at all — there is no var() bridge into filter internals. The only two ways to animate them are JS setAttribute (used by both demos here) and SMIL <animate> targeting the primitive element directly.

2. Browser internals

Applying a filter forces the browser to rasterize the affected element into an offscreen buffer, run the filter's pixel-level operations against it (a blur convolution; noise generation and remapping for turbulence/displacement), then composite the result back into the page. This is a fundamentally different and more expensive pipeline than anything else in this handbook — not paint-only (Path Drawing's stroke-dashoffset), not geometry-only (Shape Morphing's d changes), but a full rasterize-and-reprocess step, repeated every frame these demos run.

3. Step-by-step implementation

Step 1 — the filter and its primitive live in a hidden <svg>, referenced from the target element via filter: url(#id):

<svg width="0" height="0" style="position:absolute">
  <filter id="blur-filter">
    <feGaussianBlur id="blur-primitive" in="SourceGraphic" stdDeviation="0" />
  </filter>
</svg>
<div id="blur-card" style="filter: url(#blur-filter)" tabindex="0">...</div>

Step 2 — animate the primitive's own attribute directly from JS, lerping toward a target each frame exactly like Phase 4's Cursor Follower did for pointer position:

import { lerp } from '../../utils/math.js';

let blurCurrent = 0;
let blurTarget = 0;

function animateBlur() {
  blurCurrent = lerp(blurCurrent, blurTarget, 0.2);
  blurPrimitive.setAttribute('stdDeviation', blurCurrent.toFixed(2));
  if (Math.abs(blurTarget - blurCurrent) > 0.05) {
    requestAnimationFrame(animateBlur);
  }
}

blurCard.addEventListener('pointerenter', () => { blurTarget = 6; animateBlur(); });
blurCard.addEventListener('pointerleave', () => { blurTarget = 0; animateBlur(); });

Step 3 — the "liquid" demo continuously wobbles feTurbulence's baseFrequency, producing an organic, undulating displacement with no user interaction required:

function animateLiquid(timestamp) {
  const wobble = Math.sin(timestamp / 800) * 0.01 + 0.02;
  turbulence.setAttribute('baseFrequency', `${wobble.toFixed(4)} 0.04`);
  requestAnimationFrame(animateLiquid);
}
requestAnimationFrame(animateLiquid);

5. Exercises

  1. Change Demo 1's BLUR_MAX from 6 to 20 and observe how much more expensive-looking (and slower to settle) the effect becomes.
  2. Add a second feColorMatrix primitive after the blur, animating its saturation attribute alongside the blur.
  3. Replace Demo 2's continuous Math.sin wobble with a value driven by the cursor's horizontal position, so the liquid effect responds to pointer movement instead of running on its own.

6. Common mistakes

  • Trying to animate a filter primitive's attribute with a CSS custom property and var() — this silently does nothing, since CSS variables cannot reach into a referenced filter's internal graph at all.
  • Forgetting the prefers-reduced-motion check on Demo 1 — unlike Phase 4's custom-property-driven demos, this one has no CSS transition backing it (JS sets the attribute directly every frame), so the sitewide CSS rule does nothing for it; it needs its own explicit check.
  • Changing feTurbulence's seed attribute every frame for "randomness" — this produces a jarring, discontinuous flicker instead of smooth motion, since each seed generates an unrelated noise pattern; animating baseFrequency smoothly (as Demo 2 does) is the way to get organic continuous motion.

7. Performance considerations

This is the most expensive category of effect in this handbook. Both demos force per-frame rasterization and pixel-level reprocessing of their target element — genuinely more costly than Phase 5a's geometry-changing d morphs, which only recompute a path's shape, not reprocess its rendered pixels. Keep filtered regions small, avoid filtering large parts of a page, and prefer triggering these effects on discrete interaction (like Demo 1's hover/focus) over always-on loops (like Demo 2) where the effect isn't essential to the page's content.

8. Accessibility considerations

Demo 1's card is focusable (tabindex="0") specifically because its blur is triggered by both :hover-equivalent pointer entry and keyboard focus (the JS listens for both pointerenter/pointerleave and focus/blur), so keyboard users get the same effect mouse users do. Demo 2's liquid effect is purely decorative and requires no interaction, so there's no keyboard-parity concern for it. Both check prefers-reduced-motion directly in JS, since neither is a CSS transition the sitewide rule could otherwise cover.

9. Mobile considerations

Demo 1's blur triggers on tap-and-hold in the same way any :hover-equivalent does on touch (inconsistently across browsers, and not on release) — an acceptable degradation for a decorative focus/hover enhancement. Demo 2's liquid effect runs identically regardless of input device, since it needs no interaction at all.
