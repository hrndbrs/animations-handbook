Path Drawing

Making an SVG stroke look like it's being drawn in real time, three different ways.

1. Theory

The "line draw" effect works by hiding the stroke behind a dash pattern exactly as long as the path itself, then sliding that dash pattern back into view. stroke-dasharray sets the length of the dash (and the gap after it); stroke-dashoffset shifts where along the path that pattern starts. Set both to the path's full length and the entire stroke is one long "dash" pushed completely out of view; animate stroke-dashoffset down to 0 and the dash slides back into place, reading as the line drawing itself in.

Getting "the path's full length" can happen two ways. The pathLength attribute lets you tell the browser to treat the path as if it were exactly 100 units long (or any number you choose), regardless of its real rendered size — so CSS can always use the same round number without knowing the path's actual geometry. path.getTotalLength() instead asks the browser for the path's real, geometry-dependent length in user units — the technique to use in JS when you'd rather measure the real thing than hardcode a normalized attribute.

2. Browser internals

Changing stroke-dashoffset doesn't change the path's geometry or bounding box — the underlying shape is identical, only which segments of it are painted visible changes. That makes it cheaper than animating actual path geometry (see the Shape Morphing chapter), but it is still a paint-invalidating property, not a compositor-only one like transform/opacity — the browser has to repaint the affected region on every frame the offset changes, it just doesn't have to re-run layout.

3. Step-by-step implementation

Step 1 (Demo 1) — measure the path's real length in JS, then set both dasharray and dashoffset to that value so the whole stroke starts hidden:

const pathLength = scrollPath.getTotalLength();
scrollPath.style.strokeDasharray = `${pathLength}`;
scrollPath.style.strokeDashoffset = `${pathLength}`;

Step 2 (Demo 1) — reveal it once, the first time it scrolls into view, reusing Phase 1's IntersectionObserver reveal pattern:

const observer = new IntersectionObserver((entries) => {
  entries.forEach((entry) => {
    if (entry.isIntersecting) {
      scrollPath.style.strokeDashoffset = '0';
      observer.unobserve(scrollPath);
    }
  });
}, { threshold: 0.4 });

observer.observe(scrollPath);

Step 3 (Demos 2 & 3) — with pathLength="100" set on the <path> element itself, CSS can hardcode a normalized dash length and do the rest with no JS:

.draw-demo__hover-path {
  stroke-dasharray: 100;
  stroke-dashoffset: 100;
  transition: stroke-dashoffset 0.6s ease;
}
.draw-demo__hover-stage:hover .draw-demo__hover-path {
  stroke-dashoffset: 0;
}

Step 4 (Demo 3) — a looping draw-and-erase uses a dasharray with an equal-length gap, then animates the offset a full pattern-length further than a single draw would, which pushes the dash back out of view on the far side instead of stopping fully drawn:

.draw-demo__loop-path {
  stroke-dasharray: 100 100;
  animation: draw-loop 3s ease-in-out infinite;
}
@keyframes draw-loop {
  0% { stroke-dashoffset: 100; }
  50% { stroke-dashoffset: 0; }
  100% { stroke-dashoffset: -100; }
}

5. Exercises

  1. Change Demo 1's threshold from 0.4 to 1 and observe how much further into view the path must scroll before it draws.
  2. Give Demo 3's loop a visible pause at fully-drawn by adjusting the keyframe percentages (e.g. hold at 0 from 40% to 60% instead of a single instant at 50%).
  3. Replace the zigzag path with a longer, more complex one (e.g. a wavy signature-style curve using cubic Béziers) and confirm all three demos still work without any JS/CSS changes.

6. Common mistakes

  • Setting stroke-dasharray without also setting a matching stroke-dashoffset starting value — the dash pattern exists but starts fully visible, so nothing appears to animate.
  • Mixing a hardcoded pixel dasharray value with the pathLength attribute — once pathLength is set, all dasharray/dashoffset values on that path are interpreted in the normalized 0-100 unit space, not real pixels, so a hardcoded pixel-length value from a different technique will be wrong.
  • Forgetting fill: none on a closed/filled shape being "drawn" — the fill renders immediately regardless of the stroke's dash animation, so the shape appears solid from frame one instead of outlining in first.

7. Performance considerations

Unlike almost every other demo in this handbook, this one is not composite-only: stroke-dashoffset is a paint property, so the browser repaints the path's stroke region on every change. This is still comparatively cheap (no layout, no geometry recalculation, just a segment of a fixed shape becoming visible/invisible), and stays fast for a handful of simultaneously drawing paths — it's specifically the *geometry* changes covered in the Shape Morphing chapter that get expensive, not this one.

8. Accessibility considerations

All three paths are purely decorative illustrations of the technique (aria-hidden="true") with no unique information conveyed beyond what the surrounding prose already states — a keyboard-only or screen-reader user loses nothing by never triggering Demo 2's hover reveal, matching the same reasoning this handbook's Hover Reveal & Image Distortion chapter (Phase 4) already established for purely decorative pointer effects. Demo 1's scroll reveal and Demo 3's loop both happen automatically regardless of input method or focus state, so there's no keyboard-parity gap for either of them.

9. Mobile considerations

Demo 1's scroll trigger and Demo 3's loop both work identically on touch devices — neither depends on a pointer at all. Demo 2's :hover won't auto-trigger on a touchscreen without a mouse/trackpad, so touch users simply see it at rest; an acceptable degradation for a purely decorative demonstration of the technique.
