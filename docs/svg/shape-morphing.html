Shape Morphing

Turning one SVG shape into another, without a library to solve the "which point maps to which" problem for you.

1. Theory

Morphing one path into another only makes sense if both paths are built from the same number of points, in the same order and command structure — a browser (or your own code) can't guess which point on shape A corresponds to which point on shape B if the two shapes have differing numbers of points. This is why arbitrary "morph any shape into any other shape" tools always do point-count matching/insertion behind the scenes; without a library, the two options are hand-authoring compatible paths (this chapter's Demos 1 and 3, both 4- and 12-point shapes respectively) or writing your own interpolation over matched point arrays (Demos 2 and 3, via utils/svg-morph.js).

2. Browser internals

Animating the d property directly (Demo 1) changes the path's actual geometry every frame — unlike Path Drawing's stroke-dashoffset, this is a paint-*and*-potentially-layout operation, since the browser must recompute the shape's fill/stroke geometry on every step, not just reveal/hide a fixed shape's segments. Chromium and Firefox implement native point-by-point interpolation for CSS d transitions, but only when both d values have the same command structure — this is exactly the same constraint as the JS-driven approach, just enforced by the browser instead of by your own code. Safari does not support animating d via CSS at all as of this writing, which is precisely why Demos 2 and 3 exist: a JS-computed d string works identically everywhere.

3. Step-by-step implementation

Step 1 (Demo 1) — set an initial d attribute directly on the path as a fallback, then let CSS override it via the animatable d property on hover:

.morph-demo__css-shape {
  transition: d 0.5s ease;
}
.morph-demo__css-stage:hover .morph-demo__css-shape {
  d: path("M50 10 L90 50 L50 90 L10 50 Z");
}

Step 2 (Demo 2) — define two same-length point arrays and interpolate between them every frame using a value that oscillates smoothly between 0 and 1:

import { interpolatePath } from '../../utils/svg-morph.js';

const SQUARE_POINTS = [[20, 20], [80, 20], [80, 80], [20, 80]];
const DIAMOND_POINTS = [[50, 10], [90, 50], [50, 90], [10, 50]];

function animateLoop(timestamp) {
  const t = (Math.sin(timestamp / 1000) + 1) / 2;
  loopPath.setAttribute('d', interpolatePath(SQUARE_POINTS, DIAMOND_POINTS, t));
  requestAnimationFrame(animateLoop);
}
requestAnimationFrame(animateLoop);

Step 3 (Demo 3) — a click-triggered, one-shot morph reuses the same helper, but drives t from an eased duration instead of a continuous oscillation, and disables its own trigger while running to prevent a second click from starting a competing animation mid-flight:

function stepIconMorph(timestamp, from, to, startTime) {
  const rawProgress = Math.min((timestamp - startTime) / MORPH_DURATION_MS, 1);
  iconPath.setAttribute('d', interpolatePath(from, to, easeInOutQuad(rawProgress)));
  if (rawProgress < 1) {
    requestAnimationFrame((next) => stepIconMorph(next, from, to, startTime));
  } else {
    iconTrigger.disabled = false;
  }
}

iconTrigger.addEventListener('click', () => {
  iconTrigger.disabled = true;
  requestAnimationFrame((timestamp) => stepIconMorph(timestamp, from, to, timestamp));
});

5. Exercises

  1. Change Demo 2's oscillation from Math.sin(timestamp / 1000) to a faster or slower divisor and observe how the perceived "breathing" speed changes.
  2. Add a third shape (e.g. a triangle re-drawn as a 4-point shape with one repeated point) and cycle Demo 2 through all three instead of just two.
  3. Swap Demo 3's easeInOutQuad for easeOutCubic (both already exported from utils/easing.js) and compare how the morph's motion feels.

6. Common mistakes

  • Interpolating between two point arrays with different lengths — interpolatePath throws immediately rather than silently producing a broken shape, precisely so this mistake fails loudly during development instead of rendering garbage.
  • Relying on Demo 1's CSS-only d transition as the only morph technique in a project that must support Safari — it silently does nothing there (the d property change simply doesn't animate), which is why Demos 2/3's JS-interpolation approach exists as the universally-working alternative.
  • Not disabling Demo 3's trigger button while its animation is running — clicking again mid-morph would start a second requestAnimationFrame chain racing the first over the same d attribute, the same class of bug Phase 3's Typing and Counter chapters hit with unguarded re-triggering.

7. Performance considerations

All three demos animate actual path geometry, which is a paint-and-potentially-layout operation, not composite-only — genuinely more expensive per frame than this handbook's transform-driven demos (Phases 2 and 4) or even Path Drawing's stroke-dashoffset technique. For a handful of small shapes like these, the cost is negligible; for many simultaneously-morphing complex paths, prefer simplifying point counts or morphing fewer shapes at once.

8. Accessibility considerations

Demo 1's shapes are purely decorative (aria-hidden="true"), and its hover-only trigger needs no keyboard equivalent for the same reason established in this handbook's Hover Reveal & Image Distortion chapter (Phase 4) — no functional content depends on it. Demo 2's loop requires no interaction and runs identically regardless of input method, and checks prefers-reduced-motion in JS (freezing at the midpoint instead of oscillating) since a CSS media query cannot stop a requestAnimationFrame loop. Demo 3 is a real <button>, fully keyboard-operable via Tab and Enter/Space, and uses aria-pressed so screen reader users get an explicit state announcement ("pressed"/"not pressed") rather than only a visual shape change.

9. Mobile considerations

Demo 1's :hover won't auto-trigger on touch without a mouse/trackpad — an acceptable degradation for a purely decorative demo. Demo 2's loop runs identically on any device, touch included. Demo 3 is a real button and works via tap exactly as it does via click, no touch-specific handling needed.
