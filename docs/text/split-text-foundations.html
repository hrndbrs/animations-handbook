Split Text Foundations

Before animating individual letters or words, you need them as individual elements — and that has to be done without breaking screen readers.

1. Theory

Most text-animation techniques (typing aside) need each character or word to be its own DOM element, so CSS/JS can animate them independently — a stagger-fade, a scramble effect, or a per-letter hover all require that granularity. The naive approach — wrapping every character in its own <span> — breaks accessibility if done carelessly: some screen readers announce the fragmented markup letter by letter ("H. E. L. L. O.") instead of reading the word normally.

The fix used throughout this handbook's text chapters: mark every generated span aria-hidden="true" (so assistive technology skips the fragmented markup entirely), and keep the original, unsplit text present in a visually-hidden element instead — visually-hidden, not display: none, so it still exists for screen readers to read normally while being invisible on screen.

2. Browser internals

Assistive technology builds its reading order from the accessibility tree, which is derived from the DOM and influenced by ARIA attributes — aria-hidden="true" removes an element (and its descendants) from that tree entirely, while leaving it fully present and rendered in the regular DOM/CSSOM. This is why the technique works: sighted users still see the per-character spans (needed for the visual animation), while screen readers never see them at all, reading the visually-hidden full-text element instead.

display: none would also hide an element from the accessibility tree — but it would hide it from sighted users too, which is the opposite of what's needed here (the visually-hidden technique needs to hide something visually while keeping it accessible, using absolute positioning and clipping instead of display: none).

3. Step-by-step implementation

Step 1 — split the element's text into individual characters, and build a visually-hidden fallback holding the original text:

export function splitText(element, { by = 'char' } = {}) {
  const originalText = element.textContent;
  const pieces = by === 'word'
    ? originalText.split(/(\s+)/).filter((piece) => piece.length > 0)
    : originalText.split('');

  const visuallyHidden = document.createElement('span');
  visuallyHidden.textContent = originalText;
  visuallyHidden.style.position = 'absolute';
  visuallyHidden.style.width = '1px';
  visuallyHidden.style.height = '1px';
  visuallyHidden.style.overflow = 'hidden';
  visuallyHidden.style.clip = 'rect(0 0 0 0)';
  visuallyHidden.style.whiteSpace = 'nowrap';

Step 2 — build one aria-hidden span per piece, and swap the element's content for the visually-hidden fallback plus the new spans:

  const fragment = document.createDocumentFragment();
  const pieceSpans = [];

  pieces.forEach((piece) => {
    const span = document.createElement('span');
    span.textContent = piece;
    span.setAttribute('aria-hidden', 'true');
    span.classList.add('split-text__piece');
    fragment.appendChild(span);
    if (piece.trim().length > 0) {
      pieceSpans.push(span);
    }
  });

  element.textContent = '';
  element.appendChild(visuallyHidden);
  element.appendChild(fragment);

  return pieceSpans;
}

Step 3 — this chapter's demo calls it, then stagger-fades each returned piece in with a per-index transition delay:

import { splitText } from '../../utils/split-text.js';

function runReveal() {
  const pieces = splitText(target, { by: 'char' });
  pieces.forEach((piece, index) => {
    piece.style.transitionDelay = `${index * 30}ms`;
  });
  void target.offsetWidth;
  requestAnimationFrame(() => {
    pieces.forEach((piece) => piece.classList.add('is-visible'));
  });
}

The void target.offsetWidth; line forces a synchronous style/layout flush, committing the CSS-declared initial state (opacity: 0 on freshly-created pieces) as an actual rendered frame before the next-frame class addition. Without it, the browser could collapse the insertion and the class change into a single style recalculation with no observable "before" state, and skip the transition entirely — since splitText() creates brand-new elements every call, there's nothing being "removed and re-added" here; the reflow's job is purely to commit the initial state before the visible one is requested.

5. Exercises

  1. Change the demo to split by word ({ by: 'word' }) instead of by character and observe how the stagger reads differently.
  2. Turn on a screen reader (VoiceOver, NVDA, or similar) and confirm the sentence is announced normally, not letter by letter.
  3. Temporarily remove the aria-hidden="true" attribute from the generated spans and re-test with a screen reader to hear the difference.

6. Common mistakes

  • Splitting text into spans without aria-hidden — screen readers may announce every character individually.
  • Using display: none for the "keep the real text somewhere" fallback instead of a visually-hidden technique — hides it from everyone, not just sighted users.
  • Forgetting the forced-reflow (void element.offsetWidth) step after creating fresh elements with a CSS-declared initial state — the browser can collapse the insertion and the next state change into one style recalculation with no observable "before" state, silently skipping the transition.

7. Performance considerations

Splitting text creates one DOM element per character (or word), which is fine for a sentence or two but should be used sparingly on very long text blocks — hundreds of spans is still cheap, but tens of thousands would add measurable DOM overhead for no visual benefit beyond a handful of visible words at a time.

8. Accessibility considerations

This chapter's entire subject is an accessibility technique: the generated spans are aria-hidden="true", and the original full sentence remains in a visually-hidden element for screen readers. Reduced motion is handled by the sitewide rule in shared/doc-page.css — no local override in this chapter.

9. Mobile considerations

The trigger is a real <button>, click/tap-driven. Splitting and animating a short sentence is inexpensive even on lower-powered mobile devices.
