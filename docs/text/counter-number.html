Counter & Number Animation

Counting up isn't just incrementing a number — the easing curve is what makes it feel intentional.

1. Theory

A naive number counter just increments a value on a timer. A good one applies an easing curve to the count's progress over a fixed duration, the same way any other animated value would use one — utils/easing.js's easeOutCubic (already built in Phase 1, used there only for a small comparison demo) is a direct fit here: it starts fast and decelerates into the final value, which reads as more deliberate than a linear count.

2. Browser internals

This is a plain requestAnimationFrame loop updating textContent every frame — no CSS animation is involved at all, since there's no way to transition a text node's numeric content with CSS alone. font-variant-numeric: tabular-nums is used so each digit occupies a fixed width, preventing the counter's overall width from jittering as digit shapes change.

3. Step-by-step implementation

Step 1 — track elapsed time against a fixed duration, converting that into a 0-1 progress value:

const startTime = performance.now();

function frame(now) {
  const elapsed = now - startTime;
  const t = Math.min(elapsed / DURATION_MS, 1);
  ...
}

Step 2 — run that progress through easeOutCubic before applying it to the target value, and write the result to the counter's text:

import { easeOutCubic } from '../../utils/easing.js';

function frame(now) {
  const elapsed = now - startTime;
  const t = Math.min(elapsed / DURATION_MS, 1);
  const easedT = easeOutCubic(t);
  const value = Math.round(easedT * TARGET_VALUE);
  counterEl.textContent = value.toLocaleString();
  if (t < 1) {
    requestAnimationFrame(frame);
  }
}

requestAnimationFrame(frame);

toLocaleString() adds thousands separators automatically, matching how numbers are normally displayed to users rather than showing a raw digit string.

5. Exercises

  1. Swap easeOutCubic for linear (also exported from utils/easing.js) and compare how the count feels without deceleration.
  2. Change TARGET_VALUE to a decimal (e.g. a currency amount) and format it with toLocaleString(undefined, { minimumFractionDigits: 2 }) instead.
  3. Add a second counter on the page, triggered by the same button, counting to a different target over a different duration.

6. Common mistakes

  • Incrementing the counter by a fixed step each frame instead of computing the value from elapsed time — ties the animation's total duration to the display's frame rate instead of a fixed, predictable duration.
  • Forgetting Math.min(elapsed / DURATION_MS, 1)'s upper clamp — without it, t could exceed 1 on a slow frame, and most easing functions (including easeOutCubic) aren't designed to produce sensible output outside [0, 1].
  • Omitting font-variant-numeric: tabular-nums — proportional digit widths make the counter's layout visibly shift as it counts, which reads as janky even though the animation logic itself is correct.
  • Not guarding against re-entry — clicking the trigger again mid-count starts a second concurrent requestAnimationFrame loop racing the first, both writing to the same counter element. Disabling the trigger button for the duration of the run (re-enabling it once t reaches 1) prevents this.

7. Performance considerations

A single textContent write per frame is inexpensive. The easing function itself (easeOutCubic) is a simple arithmetic expression, not a loop, so evaluating it every frame adds negligible cost.

8. Accessibility considerations

The trigger is a real <button>. The counter text updates rapidly during the animation; like the typewriter effect in the Typing & Scramble chapter, this content is not marked aria-live, so screen readers won't announce every intermediate value (which would be overwhelming) — only the final settled value is available to be read on demand. Reduced motion is handled by the sitewide rule in shared/doc-page.css for CSS transitions/animations, though — like the typewriter effect — this counter's motion is JS/rAF-driven and isn't directly affected by that CSS rule; checking matchMedia before running the eased count (and jumping straight to the final value if reduced motion is preferred) is a good exercise extension.

9. Mobile considerations

The trigger is click/tap-driven. A single-element rAF loop like this one is inexpensive enough to run smoothly even on lower-powered mobile devices.
