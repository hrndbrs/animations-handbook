Typing & Scramble Effects

Two ways to reveal text over time: growing it character by character, or resolving it out of randomness.

1. Theory

A typewriter effect reveals text by progressively growing a substring of the final string — no DOM splitting needed, since the whole thing is one text node whose content changes over time. A scramble effect is the opposite in structure: it needs every character as its own element, since each character independently cycles through random values before settling on its real one at a (usually staggered) time.

Both are timed with requestAnimationFrame rather than setInterval, tracking elapsed milliseconds between frames rather than assuming a fixed frame rate — this keeps the reveal speed consistent whether the display refreshes at 60Hz, 120Hz, or drops frames under load.

2. Browser internals

Repeatedly setting textContent on every animation frame (as the typewriter effect does) is cheap: it's a single text node update, triggering layout only for that node's line box, not a page-wide reflow. The scramble effect's many independent per-frame textContent writes (one per character, each on its own timer) are similarly cheap individually, though a very long scrambled string means many simultaneous rAF callbacks running per frame — still inexpensive at the scale of a sentence, but worth being mindful of at paragraph scale.

3. Step-by-step implementation

Step 1 — the typewriter loop tracks elapsed time since the last character was revealed, not frame count:

function runTyping() {
  let charIndex = 0;
  let lastTime = null;

  function step(timestamp) {
    if (lastTime === null) lastTime = timestamp;
    const elapsed = timestamp - lastTime;
    if (elapsed >= MS_PER_CHAR) {
      charIndex += 1;
      lastTime = timestamp;
      typingTarget.textContent = TYPING_TEXT.slice(0, charIndex);
    }
    if (charIndex < TYPING_TEXT.length) {
      requestAnimationFrame(step);
    }
  }

  typingTarget.textContent = '';
  requestAnimationFrame(step);
}

Step 2 — the blinking caret is pure CSS, driven by a step-end keyframe (an instant on/off flicker, not a smooth fade):

.typing-demo__caret {
  animation: typing-caret-blink 1s step-end infinite;
}

@keyframes typing-caret-blink {
  50% { opacity: 0; }
}

Step 3 — the scramble effect splits the target into per-character spans, then gives each one its own independent random-then-resolve timer, staggered by index:

import { splitText } from '../../utils/split-text.js';

function runScramble() {
  const pieces = splitText(scrambleTarget, { by: 'char' });
  pieces.forEach((piece, index) => {
    const finalChar = piece.textContent;
    const resolveAt = 300 + index * 40;
    let elapsed = 0;
    let lastTime = null;

    function frame(timestamp) {
      if (lastTime === null) lastTime = timestamp;
      elapsed += timestamp - lastTime;
      lastTime = timestamp;
      if (elapsed >= resolveAt) {
        piece.textContent = finalChar;
        return;
      }
      piece.textContent = randomChar();
      requestAnimationFrame(frame);
    }

    requestAnimationFrame(frame);
  });
}

Each character captures its own finalChar before scrambling starts (from splitText's returned span, whose textContent is still the real character at that point), so it knows what to resolve to once its own timer expires.

6. Common mistakes

  • Using setInterval for the typewriter effect instead of requestAnimationFrame with elapsed-time tracking — runs out of sync with the display's actual repaint and doesn't self-throttle under load.
  • Forgetting to capture each character's final value before scrambling starts — if you read textContent after scrambling has already begun, you'll capture a randomized character instead of the real one.
  • Re-running the typewriter effect without resetting typingTarget.textContent = '' first — leftover text from a prior run stays visible underneath the new reveal.
  • Not guarding against re-entry — clicking the trigger again before the typewriter finishes starts a second concurrent requestAnimationFrame loop that races the first, both writing to the same textContent. Disabling the trigger button for the duration of the run (re-enabling it once the loop completes) prevents this.

7. Performance considerations

Both effects are cheap at sentence scale: a handful of textContent writes per frame, no layout-affecting properties involved. Scramble effects scale with character count squared in perceived complexity (more characters means more simultaneous independent timers), so keep scrambled strings short for the effect to read clearly rather than as visual noise.

8. Accessibility considerations

The typewriter effect updates a single text node's content — screen readers reading it live would hear it change repeatedly as it types, which can be noisy; consider pairing rapidly-updating text like this with aria-live="off" (the default) on the container rather than announcing every intermediate state. The scramble demo's intermediate random characters are marked aria-hidden="true" by split-text.js, and its visually-hidden fallback holds the final resolved text, so screen readers hear the real word directly rather than the scrambling process. Reduced motion is handled by the sitewide rule in shared/doc-page.css — no local override in this chapter, though note that rule only shrinks CSS transition/animation durations; both effects here are pure JS/rAF-driven and are not affected by the CSS reduced-motion rule directly (a further refinement — explicitly checking matchMedia in JS before starting either effect — is a good exercise, left as Exercise 1's natural extension).

9. Mobile considerations

Both triggers are real <button> elements, click/tap-driven. Neither effect is expensive enough to be a concern on lower-powered mobile devices at sentence length.
