Letter/Word Reveal & Stagger

Combining split-text.js with scroll-triggered reveal, at the word level instead of the whole-element level.

1. Theory

Phase 1's Scroll Reveal chapter staggers whole list items in as they enter the viewport. This chapter applies the same idea one level finer: instead of revealing whole elements, it reveals individual words within a single paragraph, using split-text.js to get one element per word to stagger.

2. Browser internals

The mechanics are identical to Phase 1's Scroll Reveal chapter: IntersectionObserver detects viewport entry asynchronously without forced-layout polling, and the actual reveal animation is a plain CSS transition on opacity/transform, staying composite-only regardless of how many words are being staggered.

3. Step-by-step implementation

Step 1 — split the paragraph by word, and give each word a staggered transition delay based on its index:

import { splitText } from '../../utils/split-text.js';

const paragraph = document.getElementById('reveal-paragraph');
const words = splitText(paragraph, { by: 'word' });

words.forEach((word, index) => {
  word.style.transitionDelay = `${index * 60}ms`;
});

Step 2 — observe the whole paragraph (not each word individually) — once it's 30% visible, reveal every word at once, letting each one's own transitionDelay create the stagger:

const observer = new IntersectionObserver(
  (entries) => {
    entries.forEach((entry) => {
      if (entry.isIntersecting) {
        words.forEach((word) => word.classList.add('is-visible'));
        observer.disconnect();
      }
    });
  },
  { threshold: 0.3 }
);

observer.observe(paragraph);

This is a one-shot reveal for the whole paragraph, so observer.disconnect() (rather than unobserve() on each word — there'd be nothing to unobserve per word, since only the paragraph itself is observed) is called once triggered.

5. Exercises

  1. Change the stagger to reveal words in random order instead of reading order, by shuffling the delay assignment.
  2. Switch { by: 'word' } to { by: 'char' } and adjust the stagger delay to something much smaller, to compare word-level vs. character-level reveal pacing.
  3. Add a second paragraph below the first, observed independently, to confirm the technique scales to multiple text blocks on one page.

6. Common mistakes

  • Observing every individual word span instead of the whole paragraph — creates unnecessary observer overhead and complicates the one-shot-reveal logic for no benefit, since the words all need to reveal together anyway.
  • Forgetting observer.disconnect() (or unobserve()) after the one-shot reveal fires — the observer keeps tracking an element that will never need to fire again.
  • Applying the stagger delay after adding the is-visible class instead of before — the delay must already be set on the element before the class change triggers the transition, or the browser may not pick it up in time.

7. Performance considerations

Splitting a paragraph by word creates one element per word rather than per character, which is significantly cheaper for longer text — a reasonable default granularity for anything beyond a short headline.

8. Accessibility considerations

split-text.js's accessibility pattern applies here directly: the word spans are aria-hidden="true", and the visually-hidden fallback holds the full paragraph text, so screen readers read it normally regardless of the visual reveal state. Reduced motion is handled by the sitewide rule in shared/doc-page.css — no local override in this chapter.

9. Mobile considerations

IntersectionObserver works identically for touch-driven scrolling. Word-level splitting keeps the element count (and therefore the animation cost) lower than character-level splitting would for the same paragraph, which matters more on lower-powered devices as paragraph length grows.
