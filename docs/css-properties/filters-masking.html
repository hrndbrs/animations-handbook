Filters & Masking

Pixel-level effects — blurring, desaturating, clipping, and masking — all animatable with CSS alone.

1. Theory

filter applies a pixel-processing effect (blur, grayscale, brightness, and more) to an element and everything painted inside it, as a single visual operation — similar to opacity in that it treats the subtree as one unit. backdrop-filter is the inverse: instead of filtering the element's own content, it filters whatever is visually behind the element, which is what produces the "frosted glass" look.

clip-path and mask both hide parts of an element, but differently: clip-path defines a hard-edged geometric region (polygon, circle, etc.) — content outside it is simply not rendered, with no partial transparency. mask instead uses another image or gradient as an opacity map, so it can produce soft, gradual edges (as in this chapter's radial-gradient-masked panel) that clip-path cannot.

2. Browser internals

Unlike opacity and transform, filter is not composite-only — applying a blur or color-matrix effect requires the GPU to actually process the element's pixels (a paint-level operation), even though the result can still be composited efficiently afterward. This makes filter more expensive to animate than opacity/transform, especially at large blur radii or over large element areas, since the processing cost scales with the number of pixels being filtered.

backdrop-filter is even more expensive: the browser must re-sample whatever is currently rendered behind the element on every frame the backdrop content changes, rather than filtering a single static layer once.

3. Step-by-step implementation

Step 1 — a button styled as a card, filter and clip-path both start at their "neutral" values:

.filter-demo__box {
  filter: grayscale(0) blur(0px);
  clip-path: polygon(0 0, 100% 0, 100% 100%, 0 100%);
  transition: filter 0.5s ease, clip-path 0.5s ease;
}

Step 2 — the pressed state applies both effects together:

.filter-demo__box[aria-pressed="true"] {
  filter: grayscale(1) blur(2px);
  clip-path: polygon(10% 10%, 90% 0, 100% 90%, 0 100%);
}
const toggle = document.getElementById('filter-toggle');

toggle.addEventListener('click', () => {
  const isPressed = toggle.getAttribute('aria-pressed') === 'true';
  toggle.setAttribute('aria-pressed', String(!isPressed));
});

Step 3 — the backdrop-filter + mask panel needs no JS at all; it's a static illustration over a patterned background:

.backdrop-demo__panel {
  background: rgba(255, 255, 255, 0.08);
  backdrop-filter: blur(8px);
  -webkit-backdrop-filter: blur(8px);
  mask-image: radial-gradient(circle, black 60%, transparent 100%);
}

The -webkit-backdrop-filter prefix is still required for Safari support even in recent versions — see the compatibility table below.

5. Exercises

  1. Change the pressed-state clip-path polygon to a different shape (e.g. a triangle) and observe the hard edge clip-path always produces, contrasted with the soft mask edge below it.
  2. Add brightness() or saturate() to the filter declaration alongside grayscale() and blur() — filters can be chained, space-separated.
  3. Remove the -webkit-backdrop-filter prefix temporarily and check DevTools' rendering in a WebKit-based browser to see the frosted effect disappear.

6. Common mistakes

  • Forgetting the -webkit-backdrop-filter prefix — Safari requires it even in current versions, unlike most other modern CSS features.
  • Animating large-radius blur() filters over large areas on low-powered devices without testing — filter cost scales with pixel area and blur radius, unlike compositor-only properties.
  • Reaching for clip-path when a soft/gradual edge is needed — it only produces hard edges; mask with a gradient is the correct tool for feathered edges.

7. Performance considerations

Both filter and backdrop-filter require actual pixel processing, not just compositing — keep blur radii and affected element sizes modest, especially for backdrop-filter, which re-samples its backdrop continuously if anything behind it is also animating.

8. Accessibility considerations

The filter toggle is a real <button> with aria-pressed. Reduced motion is handled by the sitewide rule in shared/doc-page.css — no local override in this chapter. The backdrop-filter panel is decorative and static (no motion), so reduced-motion doesn't apply to it.

9. Mobile considerations

The filter toggle is click/tap-driven. backdrop-filter is well supported on mobile Safari and Chrome, though heavier blur radii can noticeably affect scroll performance on older devices if the filtered element is large or frequently repainted.
