CSS Custom Properties & @property

Why animating a CSS variable sometimes does nothing — and the at-rule that fixes it.

1. Theory

A plain CSS custom property (--my-value: ...) has no declared type — the browser treats its value as an opaque token string with type * (the CSS-wide "any" type). This means the browser has no idea how to compute an in-between value when the property changes, so it cannot animate it: the value snaps directly from old to new with no interpolation, even inside a transition.

@property exists to fix exactly this: it lets you register a custom property with a declared syntax (e.g. '<color>', '<length>', '<number>'), an inheritance behavior, and an initial value. Once registered, the browser knows the property's type and can interpolate it exactly like a built-in property — which is what makes it possible to smoothly animate a custom property's value directly.

2. Browser internals

Without @property, a custom property is stored and substituted at compute-time purely as text — var() performs what is effectively a textual substitution into the property that references it. There's no type information carried along, so the animation engine (which needs to know "this is a color, interpolate its RGB channels" or "this is a length, interpolate the number") has nothing to work with.

@property moves the custom property into the same typed-value system built-in CSS properties already use, which is also what enables browsers to validate its value (an invalid value for the declared syntax falls back to initial-value instead of silently being accepted as an opaque string).

3. Step-by-step implementation

Step 1 — register a typed custom property for color:

@property --swatch-color {
  syntax: '<color>';
  inherits: false;
  initial-value: #4f46e5;
}

Step 2 — the typed swatch transitions the custom property itself directly, and its background reads that property via var():

.custom-properties-demo__swatch--typed {
  background: var(--swatch-color, #4f46e5);
  transition: --swatch-color 0.6s ease;
}

.custom-properties-demo__toggle[aria-pressed="true"] ~ .custom-properties-demo__row .custom-properties-demo__swatch--typed {
  --swatch-color: #16a34a;
}

Step 3 — the untyped swatch looks almost identical, but its custom property is never registered with @property, so its transition: --swatch-color-untyped 0.6s ease declaration has no interpolation effect — the color still changes, but instantly rather than smoothly:

.custom-properties-demo__swatch--untyped {
  --swatch-color-untyped: #4f46e5;
  background: var(--swatch-color-untyped);
  transition: --swatch-color-untyped 0.6s ease;
}

Step 4 — since not every browser supports @property yet, feature-detect it via its companion JS API, CSS.registerProperty, and hide the typed comparison (showing a note instead) if unsupported:

const supportsAtProperty = typeof CSS !== 'undefined' && typeof CSS.registerProperty === 'function';

if (!supportsAtProperty) {
  document.querySelector('.custom-properties-demo').classList.add('no-at-property-support');
  document.getElementById('at-property-unsupported-note').hidden = false;
}

Step 5 — as numeric proof the two swatches actually behave differently, sample both elements' computed background color across a few animation frames after each toggle:

function sampleColors() {
  const samples = [];
  let frame = 0;
  function tick() {
    const untypedColor = getComputedStyle(untypedSwatch).backgroundColor;
    const typedColor = getComputedStyle(typedSwatch).backgroundColor;
    samples.push(`frame ${frame}: untyped=${untypedColor}  typed=${typedColor}`);
    frame += 1;
    if (frame < 5) {
      requestAnimationFrame(tick);
    } else {
      output.textContent = samples.join('\n');
    }
  }
  requestAnimationFrame(tick);
}

Reading the sampled output after clicking the toggle: the untyped swatch's color reaches its final value on frame 0 (no intermediate values), while the typed swatch's color visibly passes through intermediate RGB values across the sampled frames.

5. Exercises

  1. Change syntax: '<color>' to syntax: '<length>' and repurpose --swatch-color to animate a width or border-radius value instead of a color.
  2. Remove the inherits: false line and add a child element inside the typed swatch that also reads var(--swatch-color), to see the registered property inherit.
  3. Try setting an invalid value (e.g. --swatch-color: not-a-color;) in DevTools and confirm it falls back to initial-value instead of being silently accepted.

6. Common mistakes

  • Expecting transition: --my-color 0.3s; to animate a plain custom property — without @property's syntax declaration, the browser has no type information and cannot interpolate it.
  • Registering @property but forgetting initial-value — it's required; omitting it makes the whole @property rule invalid and the property behaves as if unregistered.
  • Assuming every browser supports @property without a fallback check — it's a relatively recent addition (Baseline widely-available only since 2024) and older browser versions still in use will silently ignore the rule.

7. Performance considerations

Interpolating a typed custom property costs about the same as interpolating any built-in property of the same type (a <color>-typed custom property interpolates like any other color) — @property doesn't introduce additional runtime cost beyond making the interpolation possible at all.

8. Accessibility considerations

The toggle is a real <button> with aria-pressed. Reduced motion is handled by the sitewide rule in shared/doc-page.css — no local override needed. The unsupported-browser fallback note is a real (non-display: none-only-visually) text element, so screen reader users on unsupported browsers still get an explanation rather than a silently missing second swatch.

9. Mobile considerations

The toggle is click/tap-driven. @property support on mobile Safari and Chrome tracks their desktop counterparts' versions, so the same feature-detection fallback covers both.
