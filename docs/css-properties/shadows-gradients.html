Shadows & Gradients

Depth cues and moving color — one via transition, one via @keyframes.

1. Theory

box-shadow and text-shadow both accept offset, blur radius, spread (box-shadow only), and color, and both can be transitioned smoothly like any other animatable property. They exist as separate properties (rather than one shadow property for everything) because a box's shadow and its text's shadow are visually and semantically distinct effects that are frequently wanted independently of each other.

Gradients (linear-gradient(), radial-gradient()) aren't a property themselves — they're values usable anywhere an image is accepted, most commonly background. Animating a gradient's motion is done via background-position (shifting where the gradient image is anchored) combined with background-size larger than 100%, so there's room for the anchor point to travel.

2. Browser internals

box-shadow is a paint-level effect, not composite-only — the browser must actually rasterize the shadow's blurred edge, which is more expensive than a pure transform/opacity change, though modern browsers cache and reuse this rendering efficiently for shadows that aren't changing shape every frame.

This chapter's gradient bar uses CSS animation (via @keyframes) rather than transition. The difference: a transition only runs in response to a property value actually changing (e.g. a class toggle), while an animation can run indefinitely on its own once applied, driven by the keyframe percentages you define rather than a before/ after state pair. This is why the gradient bar needs no JS at all — animation-iteration-count: infinite keeps it running forever without any state to toggle.

3. Step-by-step implementation

Step 1 — the shadow card starts with a subtle resting shadow, no text-shadow:

.shadow-demo__card {
  box-shadow: 0 2px 6px rgba(0, 0, 0, 0.3);
  text-shadow: 0 0 0 transparent;
  transition: box-shadow 0.4s ease, transform 0.4s ease, text-shadow 0.4s ease;
}

Step 2 — the pressed state lifts the card (via transform: translateY(), reusing Chapter 1's technique) and deepens both shadows:

.shadow-demo__card[aria-pressed="true"] {
  box-shadow: 0 20px 40px rgba(79, 70, 229, 0.45);
  transform: translateY(-6px);
  text-shadow: 0 0 12px rgba(255, 255, 255, 0.8);
}
const toggle = document.getElementById('shadow-toggle');

toggle.addEventListener('click', () => {
  const isPressed = toggle.getAttribute('aria-pressed') === 'true';
  toggle.setAttribute('aria-pressed', String(!isPressed));
});

Step 3 — the gradient bar: a 3-stop gradient stretched to 200% width, so half of it is always off to one side, ready to scroll into view:

.gradient-demo__bar {
  background: linear-gradient(90deg, var(--color-accent), var(--color-accent-2), var(--color-accent));
  background-size: 200% 100%;
  animation: gradient-shift 4s linear infinite;
}

@keyframes gradient-shift {
  from { background-position: 0% 50%; }
  to { background-position: 200% 50%; }
}

5. Exercises

  1. Change the gradient bar's animation-timing-function from linear to ease-in-out and observe how the constant-speed scroll becomes a back-and-forth pulse in perceived speed (even though it's still moving one direction).
  2. Add a colored box-shadow that matches the gradient's accent color, animated with the same @keyframes rule, to combine both techniques.
  3. Reduce background-size from 200% 100% to 150% 100% and observe how the gradient's travel distance (and therefore its perceived speed at the same duration) changes.

6. Common mistakes

  • Using transition for an effect that should loop forever — transition only ever runs once per state change; continuous motion needs animation with animation-iteration-count: infinite.
  • Forgetting background-size larger than 100% when animating background-position on a gradient — without extra room, there's nothing for the position to travel through.
  • Stacking many large-blur box-shadows on frequently-animated elements — each shadow layer adds paint cost; prefer one well-tuned shadow over several redundant ones.

7. Performance considerations

box-shadow/text-shadow transitions are paint-level, not composite-only, so keep blur radii reasonable on elements that animate frequently. The gradient bar's background-position animation is comparatively cheap since it's a single element with no blur.

8. Accessibility considerations

The shadow card is a real <button> with aria-pressed. The gradient bar is marked role="img" with a descriptive aria-label noting it's decorative, since it conveys no information assistive technology needs to read as text. Reduced motion for both is handled by the sitewide rule in shared/doc-page.css — for the infinite gradient animation specifically, the global rule's animation-iteration-count: 1 !important stops it from looping continuously, and the near-zero duration keeps that single iteration effectively instant.

9. Mobile considerations

The shadow card is click/tap-driven. The gradient bar requires no interaction at all, so it behaves identically on any device; keeping it to a single non-blurred gradient layer keeps its cost low on lower-powered mobile GPUs.
