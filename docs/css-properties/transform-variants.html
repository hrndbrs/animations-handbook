Transform Variants

Chapter 1 covered one transform (rotateY). This chapter covers the rest of the family.

1. Theory

transform accepts several distinct function types, each moving/reshaping an element differently: translate() shifts position, rotate() spins around a point, scale() resizes, and skew() shears the element's angles. All of them share the same performance characteristic covered in Chapter 1: none affect layout, so all can run on the compositor.

transform-origin sets the point functions like rotate() and scale() pivot around — default is the element's center. perspective, applied to a parent, gives 3D transforms (rotate3d(), translate3d()) a vanishing point to project against, which is what makes a 3D rotation look like it has depth instead of just flattening.

Multiple transform functions can be combined in one transform value, applied left to right — the order matters, since each function transforms the coordinate system the next one operates in (e.g. translateX(50px) rotate(45deg) produces a different result than rotate(45deg) translateX(50px), because the second example's translation happens along the already-rotated axis).

2. Browser internals

Every function in a transform value is ultimately combined into a single transformation matrix before the browser applies it. Because the whole combined effect is one matrix operation on the compositor, combining four transform functions costs no more, per frame, than applying just one — the compositor doesn't re-run four separate passes.

perspective works by altering how the browser projects 3D coordinates onto the 2D screen: a smaller perspective value creates a more dramatic, closer vanishing point (elements further along the z-axis shrink faster), while a larger value approaches a flat, undistorted projection.

3. Step-by-step implementation

Step 1 — the box's transform is composed from four CSS custom properties, each with a fallback default:

.transform-demo__box {
  transform: translateX(var(--tx, 0px)) rotate(var(--rot, 0deg)) scale(var(--scl, 1)) skew(var(--skx, 0deg));
  transition: transform 0.15s ease-out;
}

Step 2 — the parent stage gets perspective, so any future 3D variant added to the box would render with depth:

.transform-demo__stage {
  perspective: 600px;
}

Step 3 — four native range inputs, one per transform function:

<input type="range" id="translate-x" min="-80" max="80" value="0">
<input type="range" id="rotate" min="-180" max="180" value="0">
<input type="range" id="scale" min="50" max="150" value="100">
<input type="range" id="skew" min="-45" max="45" value="0">

Step 4 — each slider's input event writes its value into the matching CSS custom property; the browser recomputes the combined transform automatically since it references those properties via var():

function update() {
  box.style.setProperty('--tx', `${translateXInput.value}px`);
  box.style.setProperty('--rot', `${rotateInput.value}deg`);
  box.style.setProperty('--scl', String(Number(scaleInput.value) / 100));
  box.style.setProperty('--skx', `${skewInput.value}deg`);
}

[translateXInput, rotateInput, scaleInput, skewInput].forEach((input) => {
  input.addEventListener('input', update, { passive: true });
});

Note this script never writes to box.style.transform directly — it only ever sets the four custom properties, and CSS (Step 1) is what actually composes them into a transform.

5. Exercises

  1. Reorder the functions in the transform declaration (e.g. put scale() first) and observe how combining rotate + translate in a different order changes the result.
  2. Add a fifth slider controlling transform-origin's horizontal position and observe how it changes where rotate()/scale() pivot from.
  3. Replace rotate() with rotate3d(1, 1, 0, var(--rot)) and increase the stage's perspective value to compare 2D vs 3D rotation.

6. Common mistakes

  • Assuming transform function order doesn't matter — translateX() rotate() and rotate() translateX() produce different results because each function transforms the coordinate space the next one operates in.
  • Setting perspective on the same element being rotated in 3D, instead of on its parent — the element needs a parent with perspective to have a vanishing point to project against.
  • Writing directly to element.style.transform from JS on every input event instead of updating custom properties — works, but couples the JS to the exact transform string, whereas the custom-property approach (this chapter's pattern) lets the CSS declare the full transform independently and JS just supplies values.

7. Performance considerations

Updating four CSS custom properties on every slider input event is cheap: the browser recomputes one combined transform matrix per frame, and since the box is composite-only (no layout-affecting properties involved), dragging any slider stays smooth even on modest hardware.

8. Accessibility considerations

Native <input type="range"> elements are keyboard-operable by default (arrow keys adjust the value once focused) and expose their current value to assistive technology without any extra ARIA. Reduced motion is handled by the sitewide rule in shared/doc-page.css — no local override needed here either, though this demo's transition: transform 0.15s is already short enough that the difference is minor compared to Chapter 1's 0.6s flip.

9. Mobile considerations

Range inputs work with touch out of the box (drag the thumb). On very small viewports, the four labeled sliders stack in a single column via flex-direction: column, keeping each slider's touch target full-width and easy to grab.
