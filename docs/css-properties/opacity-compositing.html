Opacity & Compositing

The simplest animatable property, and a surprisingly deep rabbit hole into how browsers composite layers.

1. Theory

opacity controls how transparent an element (and everything painted inside it) is, from 0 (invisible) to 1 (fully opaque). Unlike background-color with an alpha channel, opacity applies to the element as a whole — its children, borders, and any nested content all fade together as one unit, rather than each being individually semi-transparent.

This exists as a distinct property (rather than requiring every color property to carry its own alpha value) because fading a whole subtree uniformly is an extremely common animation need — crossfades, fade-in reveals, disabled-state dimming — and because, as covered next, it can be animated far more cheaply than semi-transparent colors applied individually to every descendant.

2. Browser internals

Any element with opacity less than 1 creates a new stacking context. A stacking context is a self-contained unit for the purposes of paint order — child elements' z-index values are only compared against siblings within the same stacking context, never against elements outside it. This is why an element with low z-index can still visually sit "on top of" a high-z-index element elsewhere in the page: if they belong to different stacking contexts, their z-index values are never compared directly.

Like transform, an animating opacity value can be handled purely on the compositor thread: the browser paints the element's content once, then the GPU blends that already-painted layer at whatever opacity the current frame calls for — no repainting of the element's internal pixels on every frame. This is why opacity and transform are the two properties most often recommended for animation: both can skip Layout and Paint entirely once an element is promoted to its own compositor layer.

3. Step-by-step implementation

Step 1 — two layers stacked on top of each other with absolute positioning:

<div class="opacity-demo__stack" id="opacity-stack">
  <div class="opacity-demo__layer opacity-demo__layer--a">Layer A</div>
  <div class="opacity-demo__layer opacity-demo__layer--b">Layer B</div>
</div>
.opacity-demo__stack {
  position: relative;
  width: 220px;
  height: 120px;
}

.opacity-demo__layer {
  position: absolute;
  inset: 0;
  transition: opacity 0.6s ease;
}

Step 2 — Layer A starts fully opaque, Layer B starts invisible:

.opacity-demo__layer--a { opacity: 1; }
.opacity-demo__layer--b { opacity: 0; }

Step 3 — a single class, toggled by JS, swaps which layer is visible; CSS transitions both simultaneously, producing the crossfade:

.opacity-demo__stack.is-crossfaded .opacity-demo__layer--a { opacity: 0; }
.opacity-demo__stack.is-crossfaded .opacity-demo__layer--b { opacity: 1; }
const toggle = document.getElementById('opacity-toggle');
const stack = document.getElementById('opacity-stack');

toggle.addEventListener('click', () => {
  const isPressed = toggle.getAttribute('aria-pressed') === 'true';
  toggle.setAttribute('aria-pressed', String(!isPressed));
  stack.classList.toggle('is-crossfaded', !isPressed);
});

5. Exercises

  1. Add a third layer and crossfade between all three by cycling through a set of classes instead of a single boolean toggle.
  2. Open DevTools' Layers panel while the crossfade runs and confirm both layers are promoted to their own compositor layers during the transition.
  3. Replace one layer's opacity transition with a visibility transition and observe why it doesn't crossfade smoothly (visibility is not interpolatable — it snaps at the halfway point of its transition duration).

6. Common mistakes

  • Assuming a low z-index child always renders behind a high z-index element elsewhere — stacking contexts (created by opacity < 1, among other triggers) reset that comparison scope.
  • Setting opacity: 0 to "hide" an element without also removing it from layout flow or interaction — the element is invisible but still occupies space and remains clickable/focusable, unlike display: none.
  • Animating individual descendant colors' alpha channels to fake a whole-subtree fade instead of using one opacity value on the parent — more properties to keep in sync, and loses the compositor-only fast path.

7. Performance considerations

Both layers in this demo only ever transition opacity, so the crossfade stays composite-only — no layout or repaint of either layer's internal content during the animation, regardless of how much content each layer contains.

8. Accessibility considerations

The toggle is a real <button> with aria-pressed, reachable and operable by keyboard. Reduced motion is handled by the sitewide rule in shared/doc-page.css (which shrinks all transition/ animation durations to near-zero) — this chapter does not define a local override, since one would be redundant.

9. Mobile considerations

The demo is click/tap-driven, not :hover-dependent, so it behaves identically on touchscreens. opacity transitions are inexpensive even on lower-powered mobile GPUs since they stay compositor-only.
