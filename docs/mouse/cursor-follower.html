Cursor Tracking & Mouse Follower

The cursor moves instantly; the element that follows it usually shouldn't.

1. Theory

The simplest way to make an element "follow" the cursor is to set its position directly to event.clientX/clientY on every pointermove event — but this makes the element rigidly glued to the cursor, which reads as mechanical rather than alive. Lerping the followed position toward the target each frame (rather than jumping straight to it) produces a trailing, catching-up motion that feels much more natural, and is the basis for most "magnetic"/"organic" cursor effects.

This is the same lerp() function from utils/math.js already used by Phase 1's Canvas Particle System chapter — the underlying idea (ease a current value toward a target every frame) is identical whether the target is a particle's home position or the live cursor position.

2. Browser internals

pointermove fires far more often than once per animation frame — on a high-polling-rate mouse it can fire well over 60 times per second. Rather than repositioning the dot directly inside the event handler, this demo only updates a plain pointer object there, and applies the actual transform change inside the requestAnimationFrame loop — decoupling "how often the raw input arrives" from "how often the DOM is actually touched," which keeps the visible update rate capped at the display's real refresh rate no matter how chatty the input events are.

3. Step-by-step implementation

Step 1 — track the raw pointer position and a separately-eased "current" position, both starting at the stage's center:

const pointer = { x: stage.clientWidth / 2, y: stage.clientHeight / 2 };
const current = { x: pointer.x, y: pointer.y };
let hasPointer = false;

Step 2 — the pointer handler only records the latest position; it never touches the DOM directly:

stage.addEventListener('pointermove', (event) => {
  const rect = stage.getBoundingClientRect();
  pointer.x = event.clientX - rect.left;
  pointer.y = event.clientY - rect.top;
  hasPointer = true;
}, { passive: true });

Step 3 — the render loop eases current toward pointer every frame, and only then applies it as a transform:

import { lerp } from '../../utils/math.js';

function animate() {
  if (hasPointer) {
    current.x = lerp(current.x, pointer.x, 0.15);
    current.y = lerp(current.y, pointer.y, 0.15);
    dot.style.transform = `translate(${current.x}px, ${current.y}px)`;
  }
  requestAnimationFrame(animate);
}

requestAnimationFrame(animate);

The third argument to lerp (0.15) is the "catch-up rate" per frame — smaller values produce a longer, more pronounced trail; values close to 1 make it nearly rigid.

5. Exercises

  1. Change the lerp factor from 0.15 to 0.05 and then to 0.5, and compare how the trail length changes.
  2. Give the dot its own independent lerped scale (grow slightly while moving, shrink at rest) using a second lerped value.
  3. Add a second, slower-following dot (a different lerp factor) to create a two-layer trailing effect.

6. Common mistakes

  • Setting the dot's position directly inside the pointermove handler instead of inside the requestAnimationFrame loop — ties visual updates to however often the raw input event fires, not to the display's actual refresh rate.
  • Forgetting to seed current's initial value — if it starts at {x: 0, y: 0} instead of a sensible default, the dot visibly flies in from the corner the first time the cursor moves.
  • Using left/top instead of transform: translate() to position the dot — forces layout on every frame instead of staying composite-only.

7. Performance considerations

Only one element updates per frame here, and only via transform, so this stays composite-only regardless of how fast the cursor moves. The pattern scales to multiple independently-lerped followers (see Exercise 3) at negligible additional cost, since each is just a couple of extra arithmetic operations per frame.

8. Accessibility considerations

The follower dot is purely decorative (aria-hidden="true") — nothing functional depends on it, so keyboard-only users lose nothing by not seeing it move. There's no keyboard equivalent for "continuous pointer position" to provide, since the effect has no functional purpose beyond visual delight. The demo checks prefers-reduced-motion in JS (CSS's media query cannot stop a requestAnimationFrame loop) and skips the lerped trailing when set, jumping the dot directly to the pointer position instead.

9. Mobile considerations

Touchscreens have no persistent hover/pointer position the way a mouse does — pointermove only fires while a finger is actively touching the screen, and pointerleave fires on lifting. The demo already handles this correctly: the dot simply doesn't move until a touch starts, and the hint text stays visible until then.
