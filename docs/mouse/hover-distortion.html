Hover Reveal & Image Distortion

Two ways an image (here, a gradient stand-in) can react to the cursor — a simple zoom, and a subtle warp.

1. Theory

A hover-zoom image effect wraps an image in an overflow: hidden frame smaller than (or equal to) the image's scaled-up size, so scaling the inner image up on hover reveals a "zoomed in" crop rather than the image visibly growing past its container's edges. This needs no JS at all — it's a plain :hover transition.

A distortion effect goes further: instead of a uniform scale, the element skews based on cursor position, producing a warping effect as if the surface were slightly pliable. This chapter uses a gradient box in place of a real photo, since this project has no image asset pipeline (the same constraint Phase 2's Opacity & Compositing / Shadows & Gradients chapters ran into for object-position) — the technique applies identically to a real <img> in its place.

2. Browser internals

Both effects only ever animate transform, so both stay composite-only regardless of how large the underlying image/element is — critically important for the zoom effect in particular, since real photos are often much larger in file size and pixel dimensions than the small decorative elements used elsewhere in this handbook.

3. Step-by-step implementation

Step 1 — the zoom frame clips anything beyond its bounds, and the inner image scales up purely via CSS transition on hover/focus:

.zoom-demo__frame {
  overflow: hidden;
}
.zoom-demo__image {
  transition: transform 0.4s ease;
}
.zoom-demo__frame:hover .zoom-demo__image {
  transform: scale(1.15);
}

Step 2 — the distortion box composes its skew from two custom properties, same pattern as every other custom-property-driven demo in this phase:

.distort-demo__box {
  --distort-x: 0deg;
  --distort-y: 0deg;
  transform: skew(var(--distort-x), var(--distort-y));
  transition: transform 0.2s ease-out;
}

Step 3 — map cursor position within the box (centered at 0) to a skew angle on each axis:

const MAX_SKEW_DEGREES = 10;

function handlePointerMove(event) {
  const rect = distortBox.getBoundingClientRect();
  const relativeX = (event.clientX - rect.left) / rect.width - 0.5;
  const relativeY = (event.clientY - rect.top) / rect.height - 0.5;
  distortBox.style.setProperty('--distort-x', `${relativeX * MAX_SKEW_DEGREES}deg`);
  distortBox.style.setProperty('--distort-y', `${relativeY * MAX_SKEW_DEGREES}deg`);
}

6. Common mistakes

  • Forgetting overflow: hidden on the zoom frame — the scaled-up image spills outside its intended bounds instead of appearing "zoomed in."
  • Setting MAX_SKEW_DEGREES too high — skew distorts text/content readability quickly; small values (5-10 degrees) read as a subtle warp, larger values just look broken.
  • Applying the zoom transform to the frame itself instead of an inner element — scales the clipping boundary along with the content, so nothing appears to zoom at all.

7. Performance considerations

Both effects are transform-only and composite-only. For a real <img>, prefer an appropriately-sized source image over relying on the browser to downscale a much larger one — the transform itself is cheap, but decoding an oversized image is not.

8. Accessibility considerations

Neither demo has a keyboard equivalent, and that's the correct call here: both are purely decorative pointer effects with no functional content of their own (the zoom frame wraps a plain gradient box, not interactive content), so there's nothing a keyboard-only user would be missing by never triggering either effect. The underlying content in both cases remains visible at rest regardless.

9. Mobile considerations

Both effects are pointer-dependent: the zoom frame's :hover and the distortion box's cursor tracking both require a mouse or trackpad. On touch devices without a pointing device, both remain at rest — an acceptable degradation for purely decorative effects with no functional content.
