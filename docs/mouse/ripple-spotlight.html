Ripple & Spotlight Effects

One effect triggered by a click (keyboard-friendly), one that only makes sense for a real pointer.

1. Theory

A Material-style ripple is a circle that expands outward from the click point and fades, giving visual confirmation that a click registered. Unlike this phase's other demos, it's triggered by click — which fires identically for a mouse click and a keyboard Enter/Space activation on a focused button — so it isn't pointer-exclusive the way continuous cursor-tracking effects are.

A spotlight effect is the opposite: it only makes sense with a real pointer, since it's meant to simulate a light source following the cursor across a surface. It's implemented almost entirely in CSS — JS only ever updates two custom properties tracking cursor position; a plain :hover rule handles showing and hiding the glow.

2. Browser internals

The ripple's <span> is a genuinely new DOM node created and destroyed per click — cheap at the rate humans click buttons, but worth explicitly removing (via the animationend listener) rather than letting ripples accumulate in the DOM indefinitely on a button clicked many times. The spotlight's radial-gradient repaints only the ::before pseudo-element's background on each pointermove, not the whole card — background changes are a paint-level operation, not layout, so this stays reasonably cheap even at high cursor-move frequency.

3. Step-by-step implementation

Step 1 — detect whether a click was keyboard-triggered (its detail is 0 for non-mouse activation) so the ripple centers sensibly instead of using a meaningless (0, 0) coordinate:

function createRipple(event) {
  const rect = rippleButton.getBoundingClientRect();
  const isKeyboardActivation = event.detail === 0;
  const clickX = isKeyboardActivation ? rect.width / 2 : event.clientX - rect.left;
  const clickY = isKeyboardActivation ? rect.height / 2 : event.clientY - rect.top;
  ...
}

Step 2 — create the ripple element sized to cover the whole button from any click point, and remove it once its animation finishes:

const ripple = document.createElement('span');
ripple.className = 'ripple-demo__ripple';
const size = Math.max(rect.width, rect.height) * 2;
ripple.style.width = `${size}px`;
ripple.style.height = `${size}px`;
ripple.style.left = `${clickX - size / 2}px`;
ripple.style.top = `${clickY - size / 2}px`;

rippleButton.appendChild(ripple);
ripple.addEventListener('animationend', () => ripple.remove());
@keyframes ripple-expand {
  to { transform: scale(1); opacity: 0; }
}
.ripple-demo__ripple {
  transform: scale(0);
  animation: ripple-expand 0.6s ease-out;
}

Step 3 — the spotlight: track the cursor via two custom properties, and let CSS's radial-gradient + :hover handle the rest:

function handleSpotlightMove(event) {
  const rect = spotlightCard.getBoundingClientRect();
  spotlightCard.style.setProperty('--spot-x', `${event.clientX - rect.left}px`);
  spotlightCard.style.setProperty('--spot-y', `${event.clientY - rect.top}px`);
}

spotlightCard.addEventListener('pointermove', handleSpotlightMove, { passive: true });
.spotlight-demo__card::before {
  background: radial-gradient(circle at var(--spot-x) var(--spot-y), rgba(79, 70, 229, 0.35), transparent 60%);
  opacity: 0;
  transition: opacity 0.3s ease;
}
.spotlight-demo__card:hover::before,
.spotlight-demo__card:focus-within::before {
  opacity: 1;
}

5. Exercises

  1. Change the ripple's color and duration, and try triggering several clicks in quick succession to confirm overlapping ripples don't interfere with each other.
  2. Change the spotlight's radial-gradient color and the 60% falloff distance, and observe how a smaller falloff produces a tighter, more focused glow.
  3. Combine both: add a ripple to the spotlight card's click as well.

6. Common mistakes

  • Assuming event.clientX/clientY are always meaningful on click — for a keyboard-triggered activation they're typically 0, which would place a ripple visibly in the wrong corner if not detected and handled.
  • Never removing old ripple elements — leaves an ever-growing pile of invisible (fully faded but still-present) <span> elements in the DOM after repeated clicks.
  • Using mousemove instead of pointermove for the spotlight — misses touch/pen input on devices where a "mouse" position can still make sense (e.g. a stylus).

7. Performance considerations

The ripple is a one-off, self-cleaning DOM node per click — no ongoing cost. The spotlight only writes two custom properties per pointermove, with the actual gradient repaint handled by the browser's compositor/paint pipeline, not JS.

8. Accessibility considerations

The ripple triggers on click, which fires identically for keyboard Enter/Space activation on a focused button — it is not pointer-exclusive. The spotlight card has tabindex="0" and a :focus-within rule alongside :hover, so keyboard focus at least reveals the glow at its default (centered) position, though the glow can't meaningfully "follow" a keyboard focus the way it follows a cursor — the card's text content remains fully readable either way, since the glow is purely decorative.

9. Mobile considerations

The ripple works identically via tap (a tap fires click just like a mouse click, with real clientX/clientY coordinates at the tap point). The spotlight has no sustained hover state on touch devices, so it simply won't show on tap-only interaction — an acceptable degradation for a decorative effect.
