Tilt Cards

A card that tilts in 3D toward wherever the cursor is within it — Phase 2's perspective work, made interactive.

1. Theory

A tilt card reads the cursor's position relative to the card's own bounds (not the page), normalized to a 0-1 range on each axis, and maps that into a rotation angle. Moving the cursor toward the card's right edge should tilt the card as if that edge is being pushed away from the viewer — which means horizontal cursor position drives rotation around the vertical (Y) axis, and vertical cursor position drives rotation around the horizontal (X) axis (the axes are perpendicular to the direction of apparent movement).

2. Browser internals

This reuses Phase 2's Transform Variants chapter's finding: a parent needs perspective for a 3D rotateX/rotateY to read as genuine depth rather than a flat squash. Here, the stage (not the card itself) carries perspective: 800px, giving the card a fixed vanishing point to rotate against as the cursor moves.

3. Step-by-step implementation

Step 1 — the card's rotation is composed from two custom properties, same pattern as the Magnetic Buttons chapter:

.tilt-demo__card {
  --tilt-x: 0deg;
  --tilt-y: 0deg;
  transform: rotateX(var(--tilt-x)) rotateY(var(--tilt-y));
  transition: transform 0.15s ease-out;
}

Step 2 — normalize the cursor's position within the card to a 0-1 range, then map it to a tilt angle centered on 0 at the card's midpoint:

const MAX_TILT_DEGREES = 12;

function handlePointerMove(event) {
  const rect = card.getBoundingClientRect();
  const relativeX = (event.clientX - rect.left) / rect.width;
  const relativeY = (event.clientY - rect.top) / rect.height;
  const tiltX = (relativeY - 0.5) * -MAX_TILT_DEGREES * 2;
  const tiltY = (relativeX - 0.5) * MAX_TILT_DEGREES * 2;
  card.style.setProperty('--tilt-x', `${tiltX}deg`);
  card.style.setProperty('--tilt-y', `${tiltY}deg`);
}

Subtracting 0.5 centers the range at the card's midpoint (so the card is flat when the cursor is exactly centered), and the sign flip on tiltX makes the top edge tilt away from the cursor when the cursor is above center, matching how a physical card would respond.

5. Exercises

  1. Increase MAX_TILT_DEGREES to 30 and observe how much more dramatic (and less subtle/premium-feeling) the effect becomes.
  2. Add a third custom property that lifts the card slightly (translateZ) while tilted, for added depth.
  3. Add a glare overlay (a semi-transparent gradient) whose position also tracks the cursor, layered on top of the card.

6. Common mistakes

  • Measuring cursor position relative to the viewport (event.clientX alone) instead of relative to the card's own bounding box — the tilt direction becomes wrong or inconsistent depending on where the card sits on the page.
  • Forgetting perspective on the stage — the rotation still applies, but reads as a flat, undistorted squash instead of genuine 3D depth.
  • Not resetting the tilt on pointerleave — the card stays frozen at its last angle instead of returning to flat when the cursor exits.

7. Performance considerations

Only transform changes, composited via custom properties — stays cheap regardless of pointer event frequency.

8. Accessibility considerations

The card's own text content remains fully readable and accessible with or without the tilt engaging — the tilt is a progressive enhancement, not a requirement for using the card. Keyboard-only users simply never see the effect, since there's no pointer position driving it; nothing about the card's content or functionality depends on the tilt having occurred.

9. Mobile considerations

A brief tilt could theoretically fire from a touch's initial contact point via pointermove during a drag, but without a hover-equivalent state, most touch interactions won't trigger sustained tilting the way a mouse does — the card simply renders flat by default on touch devices, which is an acceptable and expected degradation for a decorative effect.
